# PAGE DESIGN

本文件不是视觉规范，也不是模块边界文档。

它只回答一件事：

`用户来到这个页面，到底要做什么，页面应该怎样让这件事变得直接。`

它与 [apps/example-vue/DESIGN.md](/E:/Github/Elysian/apps/example-vue/DESIGN.md) 的关系是：

- `DESIGN.md` 负责视觉与交互约束
- 本文件负责按用户使用路径整理页面设计优化

## 1. 设计出发点

当前示例应用不缺组件，也不缺信息，真正缺的是：

- 用户一进页面能不能马上知道自己该做什么
- 页面是不是先服务任务，再展示信息
- 同一件事有没有被重复说三遍

所以本文件不按“页面长什么样”来写，而按“用户要完成什么任务”来写。

## 2. 用户总路径

用户进入后台后，真实路径非常简单：

1. 进入系统
2. 找到模块
3. 对当前对象执行操作
4. 必要时查看详情或继续下一条

因此页面设计的目标不是“把信息补全”，而是“把这四步变短”。

## 3. 页面设计优化

### 3.1 登录 / 会话进入页

用户来这里不是为了理解系统结构，而是为了进去。

用户第一眼应该看到：

- 登录入口
- 当前会话状态
- 失败反馈

用户第一步应该做：

- 输入身份信息或恢复会话

页面应该怎样组织：

- 中心单任务区域
- 主按钮明确
- 错误反馈贴着输入区

这页应该删掉：

- 后台导航
- 工作区介绍
- 与登录无关的说明文本

### 3.2 壳层导航页

用户在壳层里只做两件事：

- 找页面
- 确认当前在哪

用户第一眼应该看到：

- 清晰的左侧导航
- 当前选中态
- 当前页面主区

用户第一步应该做：

- 切到目标模块，而不是阅读欢迎信息

页面应该怎样组织：

- 导航保持稳定
- 顶栏紧凑
- 主区直接进入当前工作页

这页应该删掉：

- hero
- 欢迎语
- 与当前模块无关的全局概览

### 3.3 标准 CRUD 页面组

适用页面：

- 用户
- 角色
- 菜单
- 部门
- 字典
- 岗位
- 配置项管理
- 通知
- 租户

用户来这些页面，本质上都是：

- 找记录
- 看记录
- 新增、编辑、删除

用户第一眼应该看到：

- 查询条件
- 主列表
- 新增入口

用户第一步应该做：

- 直接筛选或直接点新增

页面应该怎样组织：

- 顶部一行查询
- 查询和重置跟着筛选走
- 列表做主体
- 详情延后到侧栏或弹层

这组页面真正要优化的不是功能多少，而是减少分心：

- 不要在列表上面堆说明卡
- 不要再加无用统计卡
- 不要把详情提前到首屏

### 3.4 客户管理页

客户页比标准 CRUD 更接近真实业务入口。

用户第一眼应该看到：

- 搜索
- 状态筛选
- 新建客户
- 列表

用户第一步应该做：

- 先找客户，或者直接新建客户

页面应该怎样组织：

- 查询工具和列表紧密相连
- 分页和排序放在列表工具层，不抢主视觉
- 编辑和删除跟着行操作走

这页要压掉的不是能力，而是噪音：

- 原型类标签
- 解释性说明
- 和分页工具重复的摘要信息

### 3.5 文件管理页

用户来文件页就是为了处理文件，不是为了理解文件系统设计。

用户第一眼应该看到：

- 搜索
- 上传入口
- 文件列表

用户第一步应该做：

- 搜索文件或上传文件

页面应该怎样组织：

- 上传动作显式
- 列表承担主要阅读任务
- 详情只在选中文件后出现

这页应该删掉：

- 说明面板
- 额外概览卡
- 和上传无关的辅助文案

### 3.6 操作日志页

用户来操作日志页，是为了回看和定位。

用户第一眼应该看到：

- 时间或关键词筛选
- 可扫读的日志列表

用户第一步应该做：

- 先缩小范围，再看具体记录

页面应该怎样组织：

- 筛选在前
- 列表在中
- 详情按需展开

这页不该往首页放的内容：

- 装饰图表
- 大段审计说明
- 与当前定位无关的统计

### 3.7 在线会话页

用户来这里，是为了判断当前会话是否正常。

用户第一眼应该看到：

- 会话列表
- 状态
- 能区分会话的关键字段

用户第一步应该做：

- 先定位异常会话

页面应该怎样组织：

- 列表优先
- 状态清楚
- 详情放在选中之后

这页应该删掉：

- 会话机制解释
- 长说明

### 3.8 流程定义页

用户来流程页，不是为了学习流程系统，而是为了找到某个流程并判断它能不能用。

用户第一眼应该看到：

- 搜索
- 流程定义列表
- 每条流程的状态和结构规模

用户第一步应该做：

- 找到目标流程

页面应该怎样组织：

- 列表作为主体
- 状态和结构规模直接跟着列表项
- 详情在选中后展开

这一页可以保留轻量结构信息，但不能回到说明式页面：

- 可保留状态分布
- 可保留节点规模
- 不保留节点结构说明文字
- 不保留版本历史类噪音标签

### 3.9 生成预览页

这是最容易堆信息的页面，但用户真正做的事情只有四步：

1. 选输入
2. 刷新预览
3. 判断能不能继续
4. 审核或应用

用户第一眼应该看到：

- 输入方式
- Schema 选择
- 主动作
- 当前是否可继续

用户第一步应该做：

- 先选输入方式和范围，再点刷新预览

页面应该怎样组织：

- 第一块只放输入
- 第二块只放状态和动作
- 第三块才放结果列表
- 详情统一放进全屏覆盖层中的居中卡片，不再使用右侧窄栏
- 若详情是高密度证据视图（如行级 diff、源码、SQL preview），优先扩大卡片内容区，而不是拆成新的侧栏

这一页最容易出现的错误是同一事实重复出现：

- 输入方式在顶部说一次，表单里再说一次
- 会话状态在摘要里说一次，按钮区再说一次
- 流程步骤写很多，但按钮本身已经表达得更清楚

所以这页的优化方向很明确：

- 砍掉顶部重复摘要
- 砍掉解释性说明句
- 让“刷新 / 审核 / 应用”直接构成页面主节奏

### 3.10 页面详情区

无论哪个模块，详情区都不该先于主任务区出现。

用户只有在已经选中对象后，才需要详情。

因此详情区的设计目标是：

- 补充，而不是抢主视觉
- 解释当前对象，而不是解释整个页面
- 内容少而准，不重复列表里已经有的事实
- 默认只给一份证据视图，不要同时摊开多份源码
- 同一流程里的事实优先合并展示，不拆成很多说明段

## 4. 文档的使用方式

以后改页面，不先问“还能加什么”，而先问：

1. 用户来这页第一步是什么
2. 页面有没有把第一步放到最前
3. 哪些信息不影响第一步，应该删
4. 哪些内容属于详情，不该出现在首屏
5. 控件本身如果已经能表达含义，就不要再补一层字段名、标题或说明句
6. 同一个对象不要同时出现在“提醒区”和“主列表”里重复展示
7. 同一时刻只给一个主动作，其他动作退成辅助动作

## 5. 逐页优化落点

这一节不写视觉细节，只写现阶段每类页面最值得优先优化的地方。

### 5.1 登录 / 会话页

现状风险：

- 容易混入系统壳层语义
- 容易在登录前就暴露模块结构

目标首屏：

- 登录表单
- 当前登录状态
- 单一主按钮

优先删除：

- 任何后台菜单影子
- 会话机制说明

### 5.2 壳层导航

现状风险：

- 容易把欢迎内容、状态信息和真实工作区混在一起

目标首屏：

- 左侧导航
- 当前页面
- 紧凑顶栏

优先删除：

- 欢迎区
- 与当前模块无关的概览

### 5.3 标准 CRUD 页面组

覆盖页面：

- `user`
- `role`
- `menu`
- `department`
- `dictionary`
- `post`
- `setting`
- `notification`
- `tenant`

现状风险：

- 查询区过重
- 列表上方堆额外说明
- 详情过早出现

目标首屏：

- 一行查询
- 主列表
- 新增按钮

优先删除：

- 说明卡
- 冗余统计
- 与当前页列表操作无关的摘要块

### 5.4 客户管理

现状风险：

- 分页与排序信息容易被重复展示
- 示例说明容易混到真实列表页里

目标首屏：

- 搜索
- 状态筛选
- 新建客户
- 客户列表

优先删除：

- 原型标签
- 预设状态类说明
- 与分页工具重复的摘要

### 5.5 文件管理

现状风险：

- 上传能力之外的辅助信息过多

目标首屏：

- 搜索
- 上传
- 文件列表

优先删除：

- 文件机制说明
- 上传区外围的额外摘要

### 5.6 操作日志

现状风险：

- 容易为了“像后台”而加过多统计信息

目标首屏：

- 时间 / 关键词筛选
- 日志列表

优先删除：

- 图表
- 总览卡
- 与定位单条日志无关的文字

### 5.7 在线会话

现状风险：

- 会话说明比会话本身更显眼

目标首屏：

- 会话列表
- 状态
- 用户或设备标识

优先删除：

- 长说明
- 机制介绍

### 5.8 流程定义

现状风险：

- 把结构说明写成页面主体
- 把只读元信息放得过多

目标首屏：

- 搜索
- 流程列表
- 状态
- 结构规模

优先删除：

- 节点结构说明文字
- 版本历史类标签
- 与选择某条流程无关的摘要

### 5.9 生成预览

现状风险：

- 输入、状态、流程、会话信息重复出现
- 步骤条和按钮同时解释同一件事
- 首屏容易被“预览说明”占满
- 容易把可选项做成“字段名 + 当前值”的表单堆叠

目标首屏：

- 输入方式
- Schema 选择
- 刷新预览
- 当前状态
- 文件级证据默认通过全屏覆盖层查看，不把长 diff 塞回右侧窄栏

优先删除：

- 顶部摘要
- 输入方式解释句
- 多处重复的会话状态
- 只读流程文字
- 与操作同行重复出现的字段标签

### 5.10 详情区

现状风险：

- 详情区提前出现，抢走列表页主任务

目标位置：

- 选中对象之后以全屏覆盖层中的居中卡片出现

优先删除：

- 页面级说明
- 与当前选中对象无关的元信息

## 6. 一句话目标

`页面不是拿来解释系统的，页面是拿来让用户更快完成当前任务的。`
