# PAGE DESIGN

本文件不是视觉规范，也不是模块边界文档。

它只回答一件事：

`用户来到这个页面，到底要做什么，页面应该怎样让这件事变得直接。`

它与 [apps/example-vue/DESIGN.md](/E:/Github/Elysian/apps/example-vue/DESIGN.md) 的关系是：

- `DESIGN.md` 负责视觉与交互约束
- 本文件负责按用户使用路径整理页面设计优化

## 1. 设计出发点

当前示例应用不缺组件，也不缺信息，真正缺的是：

- 用户一进页面能不能马上知道自己该做什么
- 页面是不是先服务任务，再展示信息
- 同一件事有没有被重复说三遍

所以本文件不按“页面长什么样”来写，而按“用户要完成什么任务”来写。

## 2. 用户总路径

用户进入后台后，真实路径非常简单：

1. 进入系统
2. 找到模块
3. 对当前对象执行操作
4. 必要时查看详情或继续下一条

因此页面设计的目标不是“把信息补全”，而是“把这四步变短”。

## 3. 页面设计优化

### 3.1 登录 / 会话进入页

用户来这里不是为了理解系统结构，而是为了进去。

用户第一眼应该看到：

- 登录入口
- 当前会话状态
- 失败反馈

用户第一步应该做：

- 输入身份信息或恢复会话

页面应该怎样组织：

- 中心单任务区域
- 主按钮明确
- 错误反馈贴着输入区

这页应该删掉：

- 后台导航
- 工作区介绍
- 与登录无关的说明文本

### 3.2 壳层导航页

用户在壳层里只做两件事：

- 找页面
- 确认当前在哪

用户第一眼应该看到：

- 清晰的左侧导航
- 当前选中态
- 当前页面主区

用户第一步应该做：

- 切到目标模块，而不是阅读欢迎信息

页面应该怎样组织：

- 导航保持稳定
- 顶栏紧凑
- 主区直接进入当前工作页

这页应该删掉：

- hero
- 欢迎语
- 与当前模块无关的全局概览

### 3.3 标准 CRUD 页面组

适用页面：

- 用户
- 角色
- 菜单
- 部门
- 字典
- 岗位
- 配置项管理
- 通知
- 租户

用户来这些页面，本质上都是：

- 找记录
- 看记录
- 新增、编辑、删除

用户第一眼应该看到：

- 查询条件
- 主列表
- 新增入口

用户第一步应该做：

- 直接筛选或直接点新增

页面应该怎样组织：

- 顶部一行查询
- 查询和重置跟着筛选走
- 列表做主体
- 详情延后到侧栏或弹层

这组页面真正要优化的不是功能多少，而是减少分心：

- 不要在列表上面堆说明卡
- 不要再加无用统计卡
- 不要把详情提前到首屏

### 3.4 客户管理页

客户页比标准 CRUD 更接近真实业务入口。

用户第一眼应该看到：

- 搜索
- 状态筛选
- 新建客户
- 列表

用户第一步应该做：

- 先找客户，或者直接新建客户

页面应该怎样组织：

- 查询工具和列表紧密相连
- 分页和排序放在列表工具层，不抢主视觉
- 编辑和删除跟着行操作走

这页要压掉的不是能力，而是噪音：

- 原型类标签
- 解释性说明
- 和分页工具重复的摘要信息

### 3.5 文件管理页

用户来文件页就是为了处理文件，不是为了理解文件系统设计。

用户第一眼应该看到：

- 搜索
- 上传入口
- 文件列表

用户第一步应该做：

- 搜索文件或上传文件

页面应该怎样组织：

- 上传动作显式
- 列表承担主要阅读任务
- 详情只在选中文件后出现

这页应该删掉：

- 说明面板
- 额外概览卡
- 和上传无关的辅助文案

### 3.6 操作日志页

用户来操作日志页，是为了回看和定位。

用户第一眼应该看到：

- 时间或关键词筛选
- 可扫读的日志列表

用户第一步应该做：

- 先缩小范围，再看具体记录

页面应该怎样组织：

- 筛选在前
- 列表在中
- 详情按需展开

这页不该往首页放的内容：

- 装饰图表
- 大段审计说明
- 与当前定位无关的统计

### 3.7 在线会话页

用户来这里，是为了判断当前会话是否正常。

用户第一眼应该看到：

- 会话列表
- 状态
- 能区分会话的关键字段

用户第一步应该做：

- 先定位异常会话

页面应该怎样组织：

- 列表优先
- 状态清楚
- 详情放在选中之后

这页应该删掉：

- 会话机制解释
- 长说明

### 3.8 流程定义页

用户来流程页，不是为了学习流程系统，而是为了找到某个流程并判断它能不能用。

用户第一眼应该看到：

- 搜索
- 流程定义列表
- 每条流程的状态和结构规模

用户第一步应该做：

- 找到目标流程

页面应该怎样组织：

- 列表作为主体
- 状态和结构规模直接跟着列表项
- 详情在选中后展开

这一页可以保留轻量结构信息，但不能回到说明式页面：

- 可保留状态分布
- 可保留节点规模
- 不保留节点结构说明文字
- 不保留版本历史类噪音标签

### 3.9 生成预览页

这是最容易堆信息的页面，但用户真正做的事情只有两段：

1. 发起一次生成
2. 处理这次生成的结果

不要把这页继续设计成“流程教学页”。

用户来这里不是来学习 generator 内部状态机，而是来完成一个任务：

- 选参数
- 生成一次预览
- 看结果
- 决定是否继续

用户第一眼应该看到：

- 新建生成
- 一组最小必要字段
- 单一主按钮
- 若已有历史结果，再在下方看到最近结果和生成结果

用户第一步应该做：

- 先完成生成配置，再点 `生成预览`

页面应该怎样组织：

- 不再展示抽象步骤条，直接用按钮表达“生成 / 审核 / 应用”
- 首屏对象固定为 `新建生成 / 最近结果 / 生成结果`
- `新建生成` 永远排在最前面，先回答“这次怎么开始”
- `新建生成` 是发起前配置，不是结果，不叫 `生成预览`
- `最近结果` 是最近一次或当前选中的参考结果，不承载表单输入
- `生成结果` 是文件级结果列表，不混入流程解释
- `新建生成` 先展示 `模块标识 / 模块名称`，再展示 `空白模板 / 复制现有模块结构 / 前端目标 / 已有文件处理方式`
- `起稿方式` 必须互斥，默认主入口是 `公共模板`；`克隆现有模块` 只作为辅助起稿方式，不与公共模板同时摊开
- `复制现有模块结构` 只表示“把已有模块结构复制到当前草稿”，不是直接重新生成那个模块
- `复制现有模块结构` 默认直接展示可点选的模块列表，不使用下拉作为首屏主入口
- 当 `克隆现有模块` 的候选会持续增长时，应优先收成搜索式入口或受控结果集，而不是把全部模块长期铺满首屏
- `Schema JSON` 退到高级输入区，默认只作为需要时展开的完整输入，不再充当首屏主入口
- 结果列表继续作为主体
- 详情统一放进全屏覆盖层中的居中卡片，不再使用右侧窄栏
- 若详情是高密度证据视图（如行级 diff、源码、SQL preview），优先扩大卡片内容区，而不是拆成新的侧栏

字段语义必须直接可懂：

- `生成来源`
  - `从空白模板开始`
  - `复制现有模块结构`
- `模块标识`
  - 生成的新模块英文标识，例如 `supplier`
- `模块名称`
  - 页面里给用户看的中文名称，例如 `供应商`
- `Schema JSON`
  - 用户真正提交的完整输入，但默认退到高级编辑区
- `前端目标`
  - 当前要生成 Vue 还是 React
- `已有文件处理方式`
  - `跳过已有文件`
  - `覆盖已有文件`
  - `仅覆盖生成文件`
  - `遇冲突停止`

页面命名必须诚实：

- 卡片标题：
  - `新建生成`
  - `最近结果`
  - `生成结果`
- 主按钮：
  - `生成预览`
  - 若已有结果，可直接显示 `重新生成预览`
- 辅助按钮：
  - `审核通过`
  - `拒绝`
  - `应用到 staging`

首屏分支必须稳定：

- `还没有生成过`
  - 首屏主对象是 `新建生成`
  - `最近结果` 只允许显示空状态或不显示
  - 主按钮只能是 `生成预览`
- `已经有结果，但还没处理完`
  - 首屏主对象仍然是 `新建生成`
  - `最近结果` 退到下面作为参考结果
  - 重新发起下一次生成时，直接在顶部配置并点击 `重新生成预览`
  - 主动作跟着结果状态走：`审核通过 / 拒绝 / 应用到 staging`
- `结果已完成`
  - 首屏仍先展示 `新建生成`
  - `最近结果` 继续作为参考
  - 下一步入口仍然放在顶部配置区
  - 不再保留“已完成后还能做什么”的说明文字

标准交互顺序必须固定：

1. 先完成 `新建生成`
2. 点击 `生成预览`
3. 页面在下方出现 `最近结果`
4. 如果有文件结果，再在 `生成结果` 中选文件查看详情
5. 根据状态执行 `审核通过 / 拒绝 / 应用到 staging`

不要反过来设计：

- 不要让用户先看到结果页，再猜怎么发起一次新的生成
- 不要让用户先理解流程图，再决定点哪个按钮
- 不要让历史结果抢在“怎么开始”前面

旧术语到页面术语的映射必须统一：

- `registered-schema` -> `参考现有模块`
- `manual-schema-json` -> `Schema JSON`
- `conflict strategy` -> `已有文件处理方式`
- `overwrite-generated-only` -> `仅覆盖生成文件`
- `fail` -> `遇冲突停止`
- `session` -> `最近结果`

如果某个词只能让研发看懂，而不能让第一次进入页面的用户看懂，就不应该直接出现在首屏。

以下词汇默认禁用：

- `人类边界`
- `AI 编辑`
- `跳过 / 覆盖 / 阻塞` 这种脱离上下文的短词
- `step / configure / review / confirm / done` 这类内部流程词
- `preview session` 这类后端实现词

原因不是这些词技术上错误，而是它们要求用户先理解实现，再理解页面。

这一页最容易出现的错误是同一事实重复出现：

- 输入方式在顶部说一次，表单里再说一次
- 会话状态在摘要里说一次，按钮区再说一次
- 流程步骤写很多，但按钮本身已经表达得更清楚
- 把“已有文件处理方式”伪装成一排没有字段名的模式按钮

所以这页的优化方向很明确：

- 砍掉顶部重复摘要
- 砍掉解释性说明句
- 让“生成 / 审核 / 应用”直接构成页面主节奏
- 让每个字段名都回答“这个控件到底在控制什么”

在继续修改真实 `generator preview` 之前，先遵守一条实现顺序：

- 所有新的页面结构试验先落到 `demohub`
- 只有当 `起稿第一页 / 结果判断页 / apply 前确认页` 三段主流程已经在 `demohub` 稳定下来，才允许迁回正式页
- 正式页负责接真实状态与真实约束，不负责继续承担试错中的信息架构探索

### 3.10 页面详情区

无论哪个模块，详情区都不该先于主任务区出现。

用户只有在已经选中对象后，才需要详情。

因此详情区的设计目标是：

- 补充，而不是抢主视觉
- 解释当前对象，而不是解释整个页面
- 内容少而准，不重复列表里已经有的事实
- 默认只给一份证据视图，不要同时摊开多份源码
- 同一流程里的事实优先合并展示，不拆成很多说明段

## 4. 文档的使用方式

以后改页面，不先问“还能加什么”，而先问：

1. 用户来这页第一步是什么
2. 页面有没有把第一步放到最前
3. 哪些信息不影响第一步，应该删
4. 哪些内容属于详情，不该出现在首屏
5. 控件本身如果已经能表达含义，就不要再补一层字段名、标题或说明句
6. 同一个对象不要同时出现在“提醒区”和“主列表”里重复展示
7. 同一时刻只给一个主动作，其他动作退成辅助动作

## 5. 逐页优化落点

这一节不写视觉细节，只写现阶段每类页面最值得优先优化的地方。

### 5.1 登录 / 会话页

现状风险：

- 容易混入系统壳层语义
- 容易在登录前就暴露模块结构

目标首屏：

- 登录表单
- 当前登录状态
- 单一主按钮

优先删除：

- 任何后台菜单影子
- 会话机制说明

### 5.2 壳层导航

现状风险：

- 容易把欢迎内容、状态信息和真实工作区混在一起

目标首屏：

- 左侧导航
- 当前页面
- 紧凑顶栏

优先删除：

- 欢迎区
- 与当前模块无关的概览

### 5.3 标准 CRUD 页面组

覆盖页面：

- `user`
- `role`
- `menu`
- `department`
- `dictionary`
- `post`
- `setting`
- `notification`
- `tenant`

现状风险：

- 查询区过重
- 列表上方堆额外说明
- 详情过早出现

目标首屏：

- 一行查询
- 主列表
- 新增按钮

优先删除：

- 说明卡
- 冗余统计
- 与当前页列表操作无关的摘要块

### 5.4 客户管理

现状风险：

- 分页与排序信息容易被重复展示
- 示例说明容易混到真实列表页里

目标首屏：

- 搜索
- 状态筛选
- 新建客户
- 客户列表

优先删除：

- 原型标签
- 预设状态类说明
- 与分页工具重复的摘要

### 5.5 文件管理

现状风险：

- 上传能力之外的辅助信息过多

目标首屏：

- 搜索
- 上传
- 文件列表

优先删除：

- 文件机制说明
- 上传区外围的额外摘要

### 5.6 操作日志

现状风险：

- 容易为了“像后台”而加过多统计信息

目标首屏：

- 时间 / 关键词筛选
- 日志列表

优先删除：

- 图表
- 总览卡
- 与定位单条日志无关的文字

### 5.7 在线会话

现状风险：

- 会话说明比会话本身更显眼

目标首屏：

- 会话列表
- 状态
- 用户或设备标识

优先删除：

- 长说明
- 机制介绍

### 5.8 流程定义

现状风险：

- 把结构说明写成页面主体
- 把只读元信息放得过多

目标首屏：

- 搜索
- 流程列表
- 状态
- 结构规模

优先删除：

- 节点结构说明文字
- 版本历史类标签
- 与选择某条流程无关的摘要

### 5.9 生成预览

现状风险：

- 输入、状态、流程、会话信息重复出现
- 步骤条和按钮同时解释同一件事
- 首屏容易被“预览说明”占满
- 容易把可选项做成“没有字段语义的一排按钮”

目标首屏：

- `新建生成`
- `当前结果`
- `生成结果`
- 主要动作只有一个：`生成预览`
- 审核与应用只挂在结果区
- 文件级证据默认通过全屏覆盖层查看，不把长 diff 塞回右侧窄栏

优先删除：

- 页面级步骤条
- 顶部摘要
- 输入方式解释句
- 多处重复的会话状态
- 只读流程文字
- `人类边界 / AI 编辑 / 阻塞` 这类要求先解释的术语

### 5.10 详情区

现状风险：

- 详情区提前出现，抢走列表页主任务

目标位置：

- 选中对象之后以全屏覆盖层中的居中卡片出现

优先删除：

- 页面级说明
- 与当前选中对象无关的元信息

## 6. 一句话目标

`页面不是拿来解释系统的，页面是拿来让用户更快完成当前任务的。`
