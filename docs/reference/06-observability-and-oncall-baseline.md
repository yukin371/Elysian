# 可观测性与值守基线

更新时间：`2026-05-06`

本文档只定义当前仓库在正式监控平台未定稿前的最小观测与值守口径。

## 当前已具备的观测入口

- `GET /health`
- `GET /metrics`
- `GET /metrics/prometheus`

## 最低要求

正式上线前，至少要满足以下 4 点：

1. 有人知道如何访问健康检查和指标端点
2. 有人知道如何查看 server 日志
3. 有人负责上线后前 30 分钟观察
4. 有人负责首日异常升级

## 上线后前 30 分钟必看信号

### 应用级

- `/health` 是否稳定返回成功
- 登录是否可用
- 关键工作区首次访问是否成功
- 5xx 是否持续出现

### 安全与权限

- 401 是否异常上升
- 403 是否异常上升
- super-admin 与 tenant admin 的关键权限边界是否异常

### 数据库

- 连接异常
- migration 失败或部分成功
- tenant 相关读写是否异常

## 指标最小口径

至少应确认以下信息能被查看或采集：

- 进程 uptime
- 进程 start time
- CPU user/system time
- memory rss / heap used

当前来源：

- `/metrics`
- `/metrics/prometheus`

## 日志最小口径

至少应确认以下日志可被查看：

- server 启动日志
- 未处理异常
- 认证失败 / 权限拒绝的关键排查信息
- workflow / tenant / generator 主链路失败时的关键错误

## 值守责任模板

上线前必须明确：

- 发布负责人：
- 首 30 分钟值守人：
- 首日值守人：
- DBA / 环境 owner：
- 异常升级路径：

## 当前未完成项

- 监控平台名称与接入方式
- 告警规则与阈值
- 通知渠道（IM / 邮件 / Pager）
- 自动化告警升级

这些项若未定稿，不应把当前仓库表述成“已具备完整生产可观测体系”。
