# 发布流转

更新时间：`2026-04-21`

本文档定义 `Elysian` 当前阶段的标准开发与发布流转，目标是让团队在不引入复杂发布平台的前提下，保持：

- `dev` 持续集成
- `main` 始终可发布
- 每次进入 `main` 的改动都有明确验证和回滚边界

---

## 分支职责

### `main`

- 默认分支
- 代表当前可发布状态
- 只接受已经在 `dev` 上稳定的改动

### `dev`

- 日常集成分支
- 功能分支、修复分支优先合并到这里
- 允许持续推进，但不允许跳过基础验证

### `feature/*`

- 新功能开发分支
- 从 `dev` 切出
- 合并目标默认是 `dev`

### `fix/*`

- 常规缺陷修复分支
- 默认从 `dev` 切出，合并回 `dev`
- 如果是线上问题，可走 hotfix 流程

### `docs/*`

- 文档、治理、流程改动分支
- 默认从 `dev` 切出，合并回 `dev`

---

## 标准节奏

当前阶段采用“持续集成 + 节点发布”的轻量节奏：

1. 日常开发全部汇入 `dev`
2. 当某一阶段目标达到可演示或可收尾状态时，冻结 `dev` 上高风险改动
3. 按发布检查清单完成验证
4. 从 `dev` 合并到 `main`
5. 合并后为 `main` 打 tag 或记录发布说明

不要求固定每周发版，但要求每次进入 `main` 都满足同一套门槛。

---

## 推荐流转

### 功能开发

1. 从 `dev` 切出 `feature/<topic>`
2. 完成功能实现、测试、文档同步
3. 本地执行：
   - `bun run check`
   - `bun run build:vue`
4. 提交 PR 到 `dev`
5. 合并后继续在 `dev` 集成验证

### 阶段发布

1. 确认 `dev` 中没有未披露的高风险改动
2. 按 [release-checklist.md](./release-checklist.md) 执行
3. 发起 `dev -> main` PR
4. 合并后记录版本说明

### 文档/治理改动

1. 从 `dev` 切出 `docs/<topic>`
2. 修改流程文档、README、路线图或 ADR
3. 走同样的 PR 流程进入 `dev`
4. 若属于发布阻塞项，可随发布一起进入 `main`

---

## Hotfix 流程

如 `main` 已存在必须立即修复的问题，使用单独的 hotfix 路径：

1. 从 `main` 切出 `fix/<topic>` 或 `hotfix/<topic>`
2. 只修复当前问题，不混入额外重构
3. 本地执行最小必要验证
4. 先合并回 `main`
5. 再把同一修复回合并到 `dev`

原则：

- hotfix 只处理阻塞问题
- hotfix 不顺带做“顺手优化”
- hotfix 合并后必须同步 `dev`

---

## 进入 `main` 的最低门槛

- `bun run check`
- `bun run build:vue`
- 相关文档已同步，或明确说明无需同步
- 风险、未验证区域、回滚路径可说明
- 不存在明显的临时代码、占位实现或脏产物

---

## GitHub 保护规则

当前 GitHub 仓库已经按本文档对应的分支职责启用保护：

### `main`

- 必须通过 Pull Request 合并，不能直接推送
- 必须通过 `validate` 状态检查
- 必须解决 review conversation 后才允许合并
- 禁止 force push
- 禁止删除分支

### `dev`

- 保留为日常集成分支
- 必须通过 `validate` 状态检查
- 不强制要求 review approval，避免把单人或高频集成流程卡死
- 禁止 force push
- 禁止删除分支

这组配置的目标不是把流程做重，而是确保：

- `main` 的进入门槛稳定且清晰
- `dev` 能持续集成，但不能跳过最基础的自动验证
- 仓库规则、CI 和团队执行方式保持一致

---

## 版本建议

当前阶段可先采用轻量版本策略：

- 阶段性里程碑：`v0.x.0`
- 小步修复：`v0.x.y`

示例：

- `v0.1.0`：P1 垂直切片闭环
- `v0.2.0`：认证与权限底座

---

## 维护原则

- 流转文档优先反映“当前真实执行方式”，不写未来理想流程
- 如果分支模型变化，优先同步本文件、`CONTRIBUTING.md` 和 `README.md`
- 如果开始引入 tag、release note、自动部署，再补 ADR 或新的发布文档
