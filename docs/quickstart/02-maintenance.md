# 维护流程（模板）

## 目标

提供最小维护闭环：问题发现 -> 修复 -> 验证 -> 收尾。

## 日常维护步骤

1. 复现问题并定位影响面。
2. 先查 owner 与边界，再按 [../DEVELOPMENT_PRINCIPLES.md](../DEVELOPMENT_PRINCIPLES.md) 判断是否该改、该抽、该扩。
3. 先写清验证方式，再改动代码。
4. 执行最小回归与全量检查。
5. 同步文档与阶段清单。
6. 提交并记录残留风险。

## 维护决策顺序

默认顺序：

1. 先边界，后实现
2. 先验证，后扩展
3. 先复用，后新增
4. 先契约，后技巧

维护时不要因为“顺手清理”而：

- 扩大本次修复范围
- 提前抽象未稳定的模式
- 新增 shared utils 或跨 owner helper
- 把局部风格统一凌驾于验证稳定性之上

## 审查时机

维护场景默认在以下时点发生审查：

1. 开始修复前先审边界、owner、复用路径、验证方式
2. 准备新增抽象、helper、adapter 或 shared util 前再审一次
3. 提交前再审验证结果、文档同步、风险和回滚路径
4. 若进入 PR，再由 reviewer 做正式审查

不要把维护 review 仅理解为“修完以后再看”。

## 回归命令

```bash
bun run check
```
