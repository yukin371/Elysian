# 架构草案

## 总体形态

建议采用 `monorepo`，按平台内核、前端适配、生成器、AI 能力、业务模板拆分。

```text
apps/
  studio/            平台控制台，负责生成、预览、配置
  example-vue/       Vue 示例前端
  example-react/     React 示例前端
  server/            Elysia 服务端
packages/
  core/              平台核心能力
  auth/              鉴权、RBAC、菜单、会话
  modules/           通用模块与业务模块约定
  schema/            实体 DSL、表单 DSL、页面 DSL
  sdk-openapi/       OpenAPI 生成客户端
  frontend-vue/      Vue 适配层
  frontend-react/    React 适配层
  generator/         代码生成器
  ai/                AI 调用、提示词、结构化输出
  templates/         代码模板
docs/
```

## 核心分层

### 1. 平台内核

由 `Elysia` 承担：

- HTTP API
- 中间件与插件装配
- OpenAPI 暴露
- 鉴权与权限校验
- SSE / streaming AI 接口
- 后台任务与事件钩子

### 2. 契约层

单一事实来源应尽量统一到 schema：

- 实体字段
- 校验规则
- 列表查询条件
- 表单布局元数据
- 权限点
- 路由元数据

契约层是代码生成和 AI 生成的共同输入。

### 3. 生成层

生成层只做两件事：

- 把结构化 schema 转成代码
- 把已有代码与模板进行安全合并

不要让生成层承担业务推理职责。

### 4. 前端适配层

前端不应直接耦合平台核心，应该定义稳定适配接口：

- 路由注册
- 页面布局
- 表单渲染
- 表格渲染
- API 调用封装
- 权限指令
- 主题和设计令牌

`Vue` 和 `React` 只是在适配层落不同实现。

## 关键架构决策

### 前端多实现，不做运行时统一渲染

不建议为了“一套描述跑两种前端”而引入复杂运行时解释器。更合理的是：

- 平台维护一套中间 DSL
- 再由模板分别生成 `Vue` 或 `React` 代码

这样生成结果更接近真实工程代码，性能和可维护性也更好。

### 首版不做 SSR 优先

后台系统和中后台业务应用的核心矛盾不是首屏 SEO，而是权限、稳定性、研发效率和模块一致性。

因此首版优先：

- SPA
- 按需懒加载
- 良好的列表性能
- 稳定的打包部署

SSR 可以作为后续扩展，而不是首发约束。

### 平台能力与业务代码分离

必须允许业务项目逐步摆脱平台模板，避免锁死。

建议把：

- 平台核心能力
- 业务模块代码
- 生成产物

分层放置，并支持局部覆写。

## 非功能要求

- 支持单体优先，保留服务拆分空间
- 默认带测试能力
- 默认带审计日志
- 默认带基础可观测能力
- 默认支持环境配置与部署脚本
- 生成结果可重复生成且可 diff
