# 2026-04-28 若依基础功能对齐矩阵

## 目标

- 用一份矩阵明确 `Elysian` 当前与“若依常用后台基础功能”之间的对齐程度。
- 固定当前优先级：先补后台基础能力与工作区闭环，再考虑继续外扩 workflow、Studio 或更重的平台能力。
- 避免把“已有后端接口”误判成“后台基础功能已经完整可用”。

## 适用边界

本文件只用于当前一轮优先级判断，覆盖：

- `apps/server` 的标准系统模块与 auth 安全基线
- `apps/example-vue` 的企业工作区装配与真实后台交互
- `packages/persistence` / `packages/schema` 中支撑上述模块的既有 owner

本文件不负责：

- 重写 `Phase 7` 之后的长期 PRD
- 定义 `generator / SQL / workflow` 的详细实施设计
- 把若依的所有扩展能力都拉进当前主线

## 对标口径

这里的“若依基本功能”只指企业后台最常用、最先被用户感知的一层能力：

- 系统管理基础模块是否闭环可用
- 登录、权限、在线会话、日志与导入导出是否具备日常使用形态
- 前端是否像一个可切换、可操作、可验证的后台，而不只是若干演示页

不要求：

- 技术栈一致
- 完整复刻若依全部模块
- 提前进入工作流扩展、低代码平台、消息中心或 SSO 体系

## 界面优先判断

本轮应采用“界面牵引、契约兜底”，而不是纯粹的“界面优先”。

原因：

- 当前后端系统模块、权限点、导出与会话治理大多已有最小闭环，继续从用户可见的后台工作流切入，更容易暴露“日常不可用”的真实缺口。
- 但界面不能反向定义新的业务真源；列表、详情、状态动作、导入导出、权限失败态仍必须落回 `apps/server` 合约与既有 canonical owner。
- 如果某个界面动作需要新增跨层能力，应先确认 owner、权限点、DTO、错误语义与验证方式，再进入实现。

实施口径：

- 优先补用户能直接感知的工作区完整度：真实路由落点、动作一致性、空态、加载态、失败态、批量操作反馈。
- 后端或 persistence 只补支撑该工作流所必需的最小契约，不借机扩成通用后台平台或第二套 shared owner。
- 对于导入链路、统计视图、文件预览、多通道通知等范围较大的能力，先用界面工作流定义最小验收场景，再决定是否进入当前主线。

## 状态说明

- `✅ 已完成`：已有实现，且当前已形成可验证闭环
- `🟡 部分完成`：已有后端或前端切片，但离“后台可用”还差关键环节
- `⚪ 未开始`：尚未进入实现
- `⛔ 当前不优先`：已有规划，但本轮不抢占第一优先级

## 当前优先顺序

### 第一优先级

1. 继续收口已有系统模块工作区，优先保证 `users / roles / menus / departments / dictionaries / settings / operation-logs / notifications / tenants` 在真实路由切换、动作一致性与失败态上更像完整后台。
2. 继续提升操作日志、通知、文件与租户等工作区的日常可用性；通知当前筛选批量已读、文件当前筛选导出与批量删除已形成小闭环，后续优先补状态反馈与后台常用交互细节。
3. 在既有导出闭环稳定后，再评估 `users / roles / menus / departments / posts / dictionaries / settings / tenants / notifications / files` 的导入链路是否值得进入当前主线。

### 第二优先级

1. 继续收口通知/公告的后台形态，但不提前进入多通道消息中心。
2. 继续提升文件、租户和数据权限工作区的日常可用性。
3. 视真实使用反馈，再决定是否需要把登录安全事件从现有 operation-log 工作区继续升级成更独立的视图或统计面板。

### 第三优先级

1. `generator / SQL` 的 Studio 化增强。
2. `workflow` 的更复杂任务语义。
3. 更重的平台化能力，如独立设计器、低代码 DSL、统一消息中心。

## 实施约束

- 优先复用既有 owner，不为了补齐功能矩阵新增第二套 shared helper 或 shared service。
- `apps/example-vue` 继续只做示例应用装配，不反向成为平台级路由框架 owner。
- workflow、generator、SQL 只做必要收口，不抢占当前第一优先级。
- 不把若依规划态功能写成“已实现事实”。

## 验证要求

- 新增后台基础能力必须至少有对应 server 合约测试或现有回归补充。
- `apps/example-vue` 的新增工作区收口不能破坏既有 `build:vue`。
- auth / session / permission 相关能力不得回退现有 401 / 403 / refresh / revoke 语义。

## 一句话结论

当前最优先的不是继续把平台做“更高”，而是先把它做“更像一个完整可用的企业后台”，优先朝若依的基础功能层对齐。
