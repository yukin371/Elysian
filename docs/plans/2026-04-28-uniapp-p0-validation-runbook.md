# 2026-04-28 uniapp P0 验证 Runbook

## 目标

- 固定 `uniapp` 在正式启动实现前的最小验证路径。
- 为 `P0-5` 提供统一执行口径，避免后续边搭环境边补验证标准。
- 明确哪些命令当前已存在，哪些命令必须等 `uniapp` app 建立后再补。

## 适用范围

- 只适用于 `uniapp` 首轮 `P0` 启动评审。
- 不代表仓库当前已经存在 `uniapp` 应用目录或可执行命令。
- 不替代后续 `P1/P2` 实现阶段的具体开发文档。

## 当前事实

### 已存在的仓库级前置能力

- 已存在 `bun run dev:vue`
- 已存在 `bun run build:vue`
- 已存在 `bun run dev:uniapp`
- 已存在 `bun run build:uniapp`
- 已存在 `bun run e2e:smoke`
- 已存在 `bun run e2e:smoke:full`
- 服务端已提供 `POST /auth/login`、`POST /auth/refresh`、`POST /auth/logout`、`GET /auth/me`

### 当前还不存在的事实

- 当前不存在 `uniapp` 自动化 E2E
- 当前不存在已验证的 Android 真机链路

说明：

当前 `uniapp` H5 骨架已经落地；本 runbook 的作用转为固定“哪些验证入口已经具备，哪些仍待补齐”。

## P0-5 目标输出

完成本 runbook 后，后续 `uniapp` 启动评审至少应补齐：

1. 一条 H5 本地调试命令
2. 一条 H5 构建命令
3. 一条 Android 真机或模拟器验证路径
4. 一份手工回归记录模板
5. 一份阻断条件判断口径

## 验证顺序

统一按以下顺序推进：

1. 先验证仓库级前置依赖
2. 再验证 H5 本地链路
3. 再验证 Android 链路
4. 最后记录差异与 blocker

不要反过来先做 Android 再补 H5，因为 H5 是最低成本的契约和页面壳验证入口。

## Step 1: 仓库级前置检查

在真正创建 `apps/example-uniapp` 之前，先确认当前主线基础未失真：

- `bun run check`
- `bun run build:vue`
- 如需后端联调基线，再执行 `bun run e2e:smoke:full`

验收标准：

- 当前主线基础命令未被破坏
- 不因为 `uniapp` 规划讨论而影响既有 `Vue` 主线稳定性

阻断条件：

- `bun run build:vue` 失败
- `bun run e2e:smoke:full` 失败且无法解释为既有已知问题

## Step 2: H5 验证入口要求

当前已补齐以下 H5 命令事实：

- 一条本地启动命令
  - 当前入口：`bun run dev:uniapp`
- 一条构建命令
  - 当前入口：`bun run build:uniapp`

### H5 最小验证项

H5 至少验证以下流程：

1. 进入登录页
2. 用户名/密码登录
3. 进入首页或个人中心
4. 调用 `/auth/me` 拉取身份
5. 进入通知列表
6. 进入通知详情
7. 执行标记已读或自动已读
8. 返回列表确认状态变化
9. 退出登录
10. 再进入页面时验证未登录态恢复逻辑

### H5 记录格式

每次 H5 验证至少记录：

- 日期
- 分支/commit
- 使用环境
- 执行命令
- 通过项
- 失败项
- 已知差异

## Step 3: Android 验证入口要求

Android 首轮只要求存在一条可重复执行的验证路径，不要求立即覆盖所有机型。

### Android 路径最低要求

后续启动时必须固定以下其中一种路径：

1. Android 模拟器路径
2. Android 真机路径

建议优先级：

- 优先真机
- 次选模拟器

原因：

- `uniapp` 首轮更关心 cookie、登录态恢复和页面交互在真实移动容器中的表现。

### Android 最小验证项

Android 与 H5 使用同一业务流，但必须额外观察：

1. 登录后 access token 是否正常进入后续请求
2. refresh cookie 是否能被后续 refresh 请求承接
3. 应用切到后台再恢复前台后，用户态是否异常丢失
4. 退出登录后是否能真正回到未登录状态

### Android 差异记录

若 Android 与 H5 行为不同，必须至少记录：

- 差异发生在登录、refresh、`me`、通知列表还是详情
- 是请求未发出、cookie 未带上、还是响应处理异常
- 是否阻断首轮范围

## Blocker 判定

以下情况视为 `P0-5` 未通过：

1. 没有 H5 本地启动命令
2. 没有 Android 可重复验证路径
3. 登录后 `/auth/me` 无法稳定拉取
4. refresh 无法在目标端稳定工作
5. 通知列表与详情无法形成最小闭环

以下情况不自动阻断，但必须记录：

1. UI 样式不完善
2. 非核心页面动效缺失
3. 首轮范围外能力未接入

## 结论

`uniapp` 的 `P0-5` 当前应收口为：

- 先守住仓库级既有验证命令
- 当前已具备 `dev:uniapp` 与 `build:uniapp` 这类 H5 本地事实
- 首轮验证按 `H5 -> Android -> 差异记录` 顺序执行
- 自动化 E2E 不作为首轮启动前置条件
