# 2026-04-28 uniapp 范围细化清单

> 执行提示：本清单只细化 `uniapp` 首轮范围，不代表已经批准进入实现。只有在 `P0` 门禁全部通过后，才允许进入 `P1/P2`。

## 目标

- 把 `uniapp` 首轮范围拆成可评审、可执行、可验收的任务清单。
- 避免后续启动时边做边改 scope。
- 固定“先门禁、后骨架、再闭环”的推进顺序。

## 总体边界

- 首轮目标：`认证 + 个人中心 + 通知`
- 首轮终端：`H5 + Android`
- 当前不做：后台工作区、多端同时铺开、复杂原生能力、跨前端 shared 抽象

## P0: 启动门禁清单

### P0-1 边界确认

- [x] 明确 `uniapp` 首轮是移动端用户自助，而不是企业后台
- [x] 明确首轮页面只包含登录、个人中心、通知列表、通知详情
- [x] 明确首轮不进入 generator / workflow / SQL / 多模块工作区
- [x] 明确首轮不承诺微信小程序等多端同时交付

验收：

- 评审时所有参与者对 In Scope / Out of Scope 口径一致

### P0-2 Owner 确认

- [x] 确认 app owner 固定为 `apps/example-uniapp`
- [x] 确认首轮暂不创建 `packages/frontend-uniapp`
- [x] 确认请求封装、登录态恢复、页面路由与通知状态先留在 app 内 owner
- [x] 确认不新增 `Vue + uniapp` 共用 adapter / state / UI owner

验收：

- `uniapp` 目录边界与 owner 说明能写进后续启动文档

### P0-3 服务端契约确认

- [x] 确认登录使用现有 `POST /auth/login`
- [x] 确认会话恢复使用现有 `POST /auth/refresh`
- [x] 确认身份信息使用现有 `GET /auth/me`
- [x] 确认通知列表、详情、已读动作使用现有通知模块 contract
- [x] 确认 `tenantCode` 作为可选输入位，可沿用默认租户语义

验收：

- 首轮不要求服务端为 `uniapp` 单独新增 API owner

### P0-4 运行时承接策略确认

- [x] 确认 `uniapp` 场景下 auth 继续兼容承接现有“内存 access token + HttpOnly refresh cookie”机制
- [x] 确认本地缓存最小策略，只覆盖登录态与必要用户摘要
- [x] 确认鉴权失效后的统一跳转策略
- [ ] 确认 H5 与 Android 之间是否存在必须提前处理的会话差异

验收：

- 能写出一页以内的“认证承接说明”，避免实现阶段反复返工

### P0-5 验证路径确认

- [x] 确认 H5 本地调试命令
- [ ] 确认 Android 真机或模拟器验证路径
- [x] 确认最小手工回归清单格式
- [x] 确认当前阶段是否不要求自动化 E2E

验收：

- 启动实现前，至少存在一条可重复执行的 H5 验证路径和一条 Android 验证路径

### P0 阻断条件

出现以下任一情况，不进入 `P1`：

- 无法确认 app owner
- 无法确认认证承接方式
- 需要为 `uniapp` 新增大量服务端特供 API
- 团队希望首轮同时覆盖多个小程序端

## P1: 最小骨架清单

### P1-1 工程骨架

- [x] 建立 `uniapp` 应用目录
- [x] 补最小启动脚本与工作区接线
- [x] 固定基础目录结构，不提前分裂过多本地模块
- [ ] 补最小环境配置说明

验收：

- 应用可启动，目录结构与 owner 一致

### P1-2 路由与页面壳

- [x] 建立登录页
- [x] 建立首页或个人中心页
- [x] 建立通知列表页
- [x] 建立通知详情页
- [x] 补最小页面跳转关系

验收：

- 页面之间可以按预期跳转，不依赖后台壳层概念

### P1-3 API 与会话骨架

- [x] 建立最小请求封装
- [x] 接入登录请求
- [x] 接入 refresh 恢复逻辑
- [x] 接入 `me` 拉取逻辑
- [x] 接入鉴权失败兜底处理

验收：

- 冷启动、登录后、登录失效后三种状态都有明确行为

### P1-4 基础状态 UI

- [ ] 补 loading 态
- [ ] 补 empty 态
- [ ] 补 error 态
- [ ] 补最小 toast 或错误提示

验收：

- 首轮页面不出现“静默失败”

## P2: 首轮业务闭环清单

### P2-1 登录闭环

- [ ] 用户可登录
- [ ] 登录后进入首页/个人中心
- [ ] 刷新或重启后能恢复最小登录态
- [ ] 退出登录后清理本地状态

验收：

- `login -> me -> logout` 完整走通

### P2-2 个人中心闭环

- [ ] 展示用户基础身份摘要
- [ ] 展示租户或身份相关必要信息
- [ ] 提供通知入口
- [ ] 提供退出登录入口

验收：

- 用户能确认“我是谁、当前是否在线、我能去哪”

### P2-3 通知列表闭环

- [x] 拉取通知列表
- [x] 显示已读/未读状态
- [x] 处理空列表态
- [x] 处理加载失败态

验收：

- 通知列表可稳定展示，不因单条异常导致整页不可用

### P2-4 通知详情 / 已读闭环

- [x] 进入通知详情
- [x] 展示通知基础内容
- [x] 支持显式标记已读，或进入详情自动已读
- [x] 已读状态能返回列表反映

验收：

- `list -> detail -> read -> list` 状态闭环成立

### P2-5 手工回归清单

- [ ] H5 验证登录/恢复/退出
- [ ] H5 验证通知列表/详情/已读
- [ ] Android 验证登录/恢复/退出
- [ ] Android 验证通知列表/详情/已读
- [ ] 记录已知差异与未覆盖项

验收：

- 至少形成一次带日期的手工回归记录

## P3: 暂缓项清单

以下项目默认不在首轮启动范围内：

- [ ] 文件上传/下载
- [ ] 个人资料编辑
- [ ] 消息推送
- [ ] 支付/地图/扫码
- [ ] workflow 待办
- [ ] 第二业务模块
- [ ] 微信小程序等多端并发适配
- [ ] 自动化 E2E 体系

说明：

这些项不是永久不做，而是必须等首轮闭环稳定后重新评审。
