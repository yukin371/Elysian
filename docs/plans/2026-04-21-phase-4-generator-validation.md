# Phase 4 Generator Validation Plan

更新时间：`2026-04-21`

> 说明：本文件是短期实施计划，只描述本轮验证范围与任务顺序。实际推进状态统一以 [../roadmap.md](../roadmap.md) 为准。

## 目标

在正式投入 `Phase 4` 之前，先验证 generator 不是只对 `customer` 这个样例成立，而是能开始沉淀为可复制的模块生成路径。

## 范围

本计划是主计划中 `P4A` 和 `P4D` 的前置验证，聚焦两件事：

- 第二个实体是否能复用现有 schema 和模板抽象
- `generated/` 到正式模块目录的 apply / merge 流程如何定义

## 本轮不做

- 完整的多实体 generator 正式发布
- Web 形式的可视化生成预览
- React 端的完整生成支持
- 三向合并的全部边界处理

## 工作包拆分

### WP-4V-1 第二实体选择

- 输出：第二个验证实体
- 推荐候选：
  - `product`
  - `order`
- 建议：优先 `product`
  - 字段和 CRUD 结构更稳定
  - 比 `customer` 多一些类型变化，但复杂度还可控
- 验收：选定一个实体并明确最小字段集

### WP-4V-2 Schema 差异验证

- 输出：第二实体 schema 与 customer schema 的差异清单
- 关注点：
  - 枚举字段
  - 可选字段
  - 数值字段
  - 搜索条件和表单控件差异
- 验收：确认现有 schema 是否需要扩展而不是硬编码 customer

### WP-4V-3 模板复用验证

- 输出：第二实体生成结果与问题清单
- 关注点：
  - server 生成是否仍遵循同一目录结构
  - Vue 列表 / 表单 / 详情模板是否有 customer 特化逻辑
  - 测试脚手架是否可以按实体名稳定输出
- 验收：生成结果能通过 `bun run check` 或明确列出阻塞项

### WP-4V-4 Apply / Merge 流程草案

- 输出：从 `generated/` 进入正式模块目录的流程定义
- 至少回答：
  - 哪些文件只允许生成
  - 哪些文件允许插入手写区块
  - 冲突时是 `skip`、`fail` 还是显式 diff
  - manifest 是否足够支撑后续更新
- 验收：形成一份可落地的 apply / merge 规则草案

### WP-4V-5 升级路径验证

- 输出：对同一实体再次生成时的行为说明
- 关注点：
  - 手写代码会不会被覆盖
  - 新字段加入后哪些文件应该自动更新
  - 删除字段时如何处理遗留代码
- 验收：至少定义“新增字段”和“已有字段变更”两条升级路径

## 建议顺序

1. 先完成 `WP-4V-1` 和 `WP-4V-2`，不要在实体和 schema 未确认前直接扩模板。
2. 再做 `WP-4V-3`，真实跑第二实体生成。
3. 基于真实结果写 `WP-4V-4` 的 apply / merge 草案。
4. 最后补 `WP-4V-5`，验证二次生成升级路径。

## 完成标准

- 第二个实体已选定并完成 schema 验证
- 现有 generator 至少能对第二实体输出一轮可检查结果
- apply / merge 规则有明确草案
- 已识别 customer 特化逻辑和模板抽象缺口

## 风险

- 如果第二实体选得过重，例如一开始就选 `order` 并带复杂子表，会把验证噪音放大
- 如果没有 apply / merge 规则，模板再多也很难进入真实开发
- 如果继续只围绕 customer 打磨，Phase 4 很容易误判为“已经成熟”
