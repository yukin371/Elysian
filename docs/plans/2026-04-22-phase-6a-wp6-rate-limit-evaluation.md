# 2026-04-22 Phase 6A WP-6 Rate Limit Evaluation

## 背景

- 当前限流实现为 `apps/server` 进程内计数（内存窗口桶）。
- 第二轮已补限流响应头观测信号：`x-ratelimit-limit` / `x-ratelimit-remaining` / `x-ratelimit-reset`。
- 在未引入 Redis / API Gateway 前，需要先明确“何时必须切换到分布式限流”。

## 目标

给出可执行的“保持现状 vs 进入分布式限流实现”判断门槛，确保扩容阶段不因单机限流导致策略失真。

## 判断门槛（进入实现前置条件）

满足以下任一条件即进入分布式限流实现评审：

1. 生产稳定运行阶段存在 `>= 2` 个 server 实例并持续一周以上。
2. 限流策略需要跨实例共享配额（同一用户/IP 在多实例间必须一致）。
3. 近 7 天内出现 `>= 3` 次“单实例命中限流但总体流量未达阈值”的误封复盘。
4. 合规或客户要求提供集中式限流审计轨迹。

## 保持现状的约束

- 单实例或低并发阶段继续使用内存限流。
- 保持 `RATE_LIMIT_*` 环境配置为唯一配置入口，不新增第二套策略源。
- 继续通过响应头和日志观察限流行为，不提前引入中间件级重构。

## 候选方案（进入实现后再二选一）

### 方案 A：Redis 计数器（优先）

- 优点：改造量可控，保留应用层策略 owner。
- 风险：需要 Redis 运维与高可用策略。

### 方案 B：网关层限流（后置）

- 优点：统一入口控制，服务侧实现更轻。
- 风险：策略可见性和业务语义映射复杂，当前仓库尚无网关 owner。

## 回滚路径

- 分布式方案上线后若出现可用性问题，允许回退到内存限流：
  - 关闭分布式限流开关（后续实现时提供）
  - 保持 `RATE_LIMIT_*` 配置继续生效
  - 复用现有 `RATE_LIMIT_EXCEEDED` 错误语义与测试

## 当前结论

- 仓库处于 `Phase 6A` 第二轮，尚未满足强制切换条件。
- 继续保留内存限流实现，优先完成 CI 稳定性观察与指标标准化。
