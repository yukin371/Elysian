# 2026-05-11 Demohub Generator Preview Prototype BDD

> 状态：计划态规格
> 范围：`apps/example-vue` 的 `demohub` workspace 内部原型
> 目标：先在 `demohub` 中验证 `generator preview` 的用户主流程，再决定如何回落到真实 workspace

## 1. 边界与假设

本文件只定义 `demohub` 中的页面原型行为，不声明功能已实现，不替代 `MODULE.md`、`DESIGN.md`、`PAGE_DESIGN.md` 或真实工作区实现说明。

本文件服务于一个明确边界：

- `demohub` 只承载原型，不接真实接口
- 原型只验证页面结构、文案层级、交互顺序和主流程判断
- 真实 DTO、session、preview report、apply evidence 和 persistence owner 仍以现有模块边界为准

当前主线仍是 generator 自举闭环与后台日常可用性打磨。因此本原型只覆盖：

- `generator preview` 的起稿阶段
- `generator preview` 的结果检查阶段
- `generator preview` 的 apply 前确认阶段

本文件不覆盖：

- 正式 API 协议变更
- 真正的生成执行或 apply 落盘
- SQL proposal 正式接入策略全文
- 其它工作区的大范围体验改造

关键假设：

- 用户默认是后台内部使用者，不需要理解 generator 内部术语才能完成主流程。
- 页面必须先表达“当前该做什么”，再暴露底层技术信息。
- 原型中结论稳定后，才能迁移到真实 workspace；未稳定前不直接改正式页。
- 如后续实现发现 owner 不清，必须先回到 `MODULE.md` 与 guardrails，而不是按本文强行落代码。

## 2. 原型目标

本原型的总目标不是“展示 generator 有多少能力”，而是：

1. 让用户更轻松地开始一次生成
2. 让用户在结果出来后立刻知道下一步该做什么
3. 让用户在 apply 前清楚知道自己为什么能继续，或为什么还不能继续

抽象后的体验原则：

1. 先结论，后证据。
2. 每步只保留一个强主动作。
3. 复杂能力后置，高级入口默认折叠。
4. 页面优先说用户语言，不优先说系统内部实现语言。
5. `demohub` 原型必须显式区分“当前只是原型验证”与“真实流程已实现”。

## 3. 用户画像

### 3.1 生成器使用者

目标：

- 快速开始新模块起稿
- 理解生成结果是否安全
- 决定是否继续审核或 apply

痛点：

- 首屏出现过多系统术语
- 被过早推进到高级 JSON 视角
- 生成完成后不知道先看哪里

### 3.2 Reviewer / 审核者

目标：

- 判断当前结果是否能继续推进
- 看清阻塞文件、变更范围和确认条件
- 决定继续、退回或重新生成

痛点：

- 证据太散
- 结果页状态不够明确
- apply 前确认页像第二份技术报告

## 4. 设计规则

### 4.1 主流程优先

每个原型页面只优先服务一个主任务，不同时承载多个平级目标。

### 4.2 高级能力后置

以下内容默认应折叠或放在次级区：

- 高级 JSON
- SQL proposal 全文
- review / apply evidence 原文
- snapshot / manifest / handoff 细节

### 4.3 先判断，再展开细节

用户进入页面后，应先得到：

- 当前状态
- 下一步建议
- 当前主动作

再决定是否阅读更细节的证据。

### 4.4 主动作唯一

每一屏最多一个强主动作；其它动作为次级或弱动作。

### 4.5 原型不接真实能力

`demohub` 中的交互只能驱动本地原型状态：

- 不生成真实 preview session
- 不写真实 staging
- 不改变真实路由、模块或 report 状态

## 5. 用户主流程

### 5.1 流程总览

用户在理想路径中应按以下顺序完成操作：

1. 进入起稿页，确定生成目标
2. 进入结果页，判断结果是否值得继续推进
3. 进入确认页，判断是否允许 apply
4. 再决定是否进入真实 apply

### 5.2 三段职责

#### `generator-start`

回答：

**这次要生成什么？**

#### `generator-review`

回答：

**这次生成结果怎么样？我下一步做什么？**

#### `generator-apply-checklist`

回答：

**我现在为什么能继续 apply，或者为什么还不能？**

## 6. 原型一：Generator Start

### 6.1 页面目标

帮助用户以最少决策成本完成起稿，而不是一开始进入高级配置或高级 JSON。

### 6.2 首屏必须出现

- 页面标题与一句简短说明
- 起稿方式选择
- 基础表单字段
- 唯一主动作“继续下一步”

### 6.3 首屏不应出现

- 大块 Schema JSON 编辑器
- 结果列表
- SQL proposal
- review/apply evidence
- 最近 session 恢复列表
- 多组并列操作按钮

### 6.4 起稿方式

- 复制现有模块
- 从模板开始
- 稍后再开高级 JSON

建议以选择卡展示，而不是普通下拉框。

### 6.5 基础字段

- 模块标识
- 模块名称
- 页面目标
- 前端目标

### 6.6 交互规则

- 选择“复制现有模块”后，不自动切换到高级 JSON。
- 选择“从模板开始”后，仍停留在表单主流程。
- 点击“继续下一步”后，应进入字段确认或页面预览阶段，而不是直接进入结果页。
- 当前页所有输入只影响本地原型状态。

### 6.7 BDD

```gherkin
Feature: Generator Start Flow
  为了让用户先明确生成目标
  作为准备新建模块的后台用户
  我希望先通过简洁表单完成起稿，而不是一开始进入高级 JSON

  Scenario: 默认进入起稿页
    Given 用户进入 generator preview
    When 页面首次加载
    Then 用户应先看到起稿方式和基础表单
    And 不应先看到高级 JSON 编辑器
    And 不应先看到结果列表或 SQL proposal

  Scenario: 选择复制现有模块
    Given 用户位于起稿页
    When 用户选择“复制现有模块”
    Then 页面应展示参考模块选择器
    And 页面不应自动切换到高级 JSON
    And 下一步应指向字段确认而不是结果页

  Scenario: 选择模板开始
    Given 用户位于起稿页
    When 用户选择“从模板开始”
    Then 页面应载入基础模板草稿
    And 页面仍保持在表单主流程
    And 高级 JSON 应保持折叠

  Scenario: 校验基础字段
    Given 用户尚未填写完整基础信息
    When 用户点击“继续下一步”
    Then 页面应提示补齐必要字段
    And 不应触发真实生成
```

## 7. 原型二：Generator Review

### 7.1 页面目标

帮助用户在生成完成后快速判断当前结果是否值得继续推进。

### 7.2 首屏必须出现

- 结果状态总结
- 文件结果列表
- 当前选中结果详情
- 下一步建议
- 唯一主动作

### 7.3 首屏不应出现

- 大段 session 元数据
- 大段 review/apply evidence 原文
- SQL proposal 全文
- 大量路径和内部技术字段
- 多个同级推进动作

### 7.4 页面结构

- 顶部状态条
- 左侧结果列表
- 右侧当前文件详情
- 底部动作区

### 7.5 状态分支

- 无阻塞且待审核
- 存在阻塞文件
- 结果不可用或失败

### 7.6 交互规则

- 默认自动选中最需要优先查看的文件。
- 阻塞文件优先于普通文件。
- 若存在阻塞，则主动作必须变为“先处理阻塞”。
- 若无阻塞但未审核，则主动作是“进入审核”。
- 技术证据默认折叠。

### 7.7 BDD

```gherkin
Feature: Generator Review Flow
  为了让用户在生成完成后快速判断结果是否可继续推进
  作为正在检查生成结果的后台用户
  我希望先看到结果结论、阻塞情况和下一步建议，而不是一开始看到过多技术细节

  Scenario: 默认进入结果页
    Given 用户已完成一次生成
    When 页面加载结果
    Then 页面应先展示结果状态总结
    And 页面应展示结果列表
    And 页面应默认选中一个最需要优先查看的文件

  Scenario: 存在阻塞文件
    Given 当前结果包含阻塞文件
    When 用户进入结果页
    Then 页面应明确提示当前不能继续 apply
    And 页面主动作应为查看阻塞文件

  Scenario: 无阻塞且待审核
    Given 当前结果没有阻塞文件
    And 当前结果尚未审核
    When 用户进入结果页
    Then 页面应提示可以继续进入审核
    And 页面主动作应为进入审核

  Scenario: 查看文件详情
    Given 用户在结果列表中选择一个文件
    When 详情区加载
    Then 页面应先展示文件概况和处理原因
    And 再展示差异摘要与源码内容
```

## 8. 原型三：Generator Apply Checklist

### 8.1 页面目标

帮助用户在 apply 前清楚知道自己是否可以继续，以及还缺什么。

### 8.2 首屏必须出现

- 当前是否可继续 apply 的结论
- 一张短确认清单
- 风险摘要
- 唯一主动作

### 8.3 首屏不应出现

- 全量结果列表
- 大段 diff
- 大段 SQL 文本
- 长篇 handoff 说明
- 多组平级强动作

### 8.4 推荐确认项

- 文件动作已确认
- 当前无阻塞冲突
- 本次仅 apply 到 staging
- SQL proposal 已确认处理方式
- 权限与菜单影响已审查
- 当前结果不是过期结果

### 8.5 状态分支

- 所有条件满足，可继续 apply
- 存在未完成项，不可 apply
- 结果可能过期，建议重新生成

### 8.6 交互规则

- 存在阻塞文件时，主动作不能是 apply。
- 未审核时，主动作应回到审核。
- 结果可能过期时，应优先建议重新生成。
- 技术证据默认不展开。

### 8.7 BDD

```gherkin
Feature: Generator Apply Checklist
  为了让用户在 apply 前清楚知道是否可以继续
  作为准备把结果应用到 staging 的后台用户
  我希望先看到简洁的确认结论和必须完成的清单，而不是再次进入复杂详情

  Scenario: 所有条件满足
    Given 当前结果已审核通过
    And 当前没有阻塞冲突
    And 当前结果仍然有效
    When 用户进入 apply checklist
    Then 页面应明确提示当前可以继续 apply
    And 页面主动作应为确认并继续 apply

  Scenario: 存在未完成项
    Given 当前结果仍有未完成确认项
    When 用户进入 apply checklist
    Then 页面应明确提示当前不能继续 apply
    And 页面应指出具体缺失项
    And 页面主动作不应为 apply

  Scenario: 结果可能过期
    Given 当前结果可能已经过期
    When 用户进入 apply checklist
    Then 页面应建议重新生成预览
    And 页面不应默认继续 apply
```

## 9. 原型迁移规则

### 9.1 可以迁移到真实 workspace 的内容

- 页面结构层级
- 按钮主次关系
- 状态提示文案
- 默认展开/折叠规则
- 文件排序优先级
- 确认项组织方式

### 9.2 不能直接从原型迁移的内容

- mock 数据形状
- 原型专用字段
- 仅用于演示的占位状态
- 未经确认的技术细节解释
- 在 `demohub` 中临时简化的内部语义

## 10. 验收标准

### 10.1 原型层验收

当以下条件满足时，认为 `demohub` 原型可进入真实实现讨论：

- 用户首屏能明确知道当前阶段要做什么
- 页面存在唯一主动作
- 复杂能力未抢占主流程
- 状态结论先于技术细节
- 不再出现“被自动推进到高级 JSON”这类违背主路径的行为

### 10.2 迁移前确认

从 `demohub` 迁回真实 workspace 前，必须补齐：

1. 哪些交互规则已经稳定
2. 哪些文案应原样迁移
3. 哪些仍只是原型假设
4. 真实 owner 应落在哪一层

### 10.3 状态覆盖面板

`demohub` 必须保留一张状态覆盖面板，作为后续正式页回灌前的轻量回归清单。该面板只展示体验状态、期望行为、迁移 owner 与覆盖状态，不接真实接口，不写 session，不替代正式自动化测试。

当前必须覆盖的状态样本：

- 起稿：复制现有模块后仍停留在表单主流程，不自动切到高级 JSON。
- 起稿：从模板开始后仍保持必要字段与下一步优先。
- 结果：存在阻塞文件时，主结论必须阻止继续 apply，并指向第一个阻塞文件。
- 结果：无阻塞且待审核时，主动作是进入审核判断，而不是 apply。
- 结果：文件详情先解释文件结论与处理原因，再展示路径、diff、源码和 SQL 证据。
- 确认：确认项已齐时，只保留可继续结论、风险摘要、短清单和唯一主动作。
- 确认：确认项缺失或结果过期时，主动作回到结果检查或重新生成。
- 壳层：直接刷新或输入 hash 时，应稳定渲染目标 workspace，不残留上一个工作区内容。

覆盖状态含义：

- `已覆盖`：已有 `demohub` 样本，并已至少一次迁回正式页或完成主路径验证。
- `待补验证`：已有 `demohub` 样本或正式页实现，但还缺稳定浏览器覆盖。
- `待设计`：已有问题或目标状态，但尚未完成原型结构设计。

## 11. 下一步建议

建议执行顺序：

1. 先在 `demohub` 中补齐 `generator-start`
2. 再补 `generator-review`
3. 再补 `generator-apply-checklist`
4. 原型稳定后，再回改真实 `generator preview` workspace
