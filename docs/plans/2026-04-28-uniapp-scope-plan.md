# 2026-04-28 uniapp 范围规划

## 目标

- 把 `uniapp` 从“第二优先级方向”推进到“可执行的范围规划”。
- 固定首轮只做最小 C 端闭环，不把 `uniapp` 规划成第二套后台或第二套平台 owner。
- 为后续是否真正启动 `uniapp` 实现，提供明确的阶段边界、验收口径与不做项。

## 当前结论

- `Vue` 仍是当前第一优先级前端主线。
- `uniapp` 作为第二优先级 C 端界面，当前只进入范围规划，不进入并行实现。
- `uniapp` 的首轮范围应收口为“移动端用户自助闭环”，而不是“复刻 `apps/example-vue` 的企业后台工作区”。

## 为什么这样收口

- 当前仓库已验证的前端主线是企业后台方向，owner 在 `apps/example-vue`。
- `uniapp` 的价值更适合落在移动端、轻交互、用户自助和多端分发，而不是再做一套后台管理壳层。
- 当前服务端已具备认证、通知、文件、租户、基础用户信息等可复用 contract，足够支撑一个小范围 C 端闭环。
- 当前并没有稳定的“小程序全家桶”验证链路，若一开始把终端铺太宽，会把问题从业务范围扩大成基础设施范围。

## 产品范围定位

### 1. 目标形态

- `uniapp` 首轮定位为“轻量 C 端应用壳”。
- 优先服务移动端用户的登录、会话恢复、消息查看、个人资料查看等高频自助动作。
- 不承担企业后台、工作区切换、复杂表单编排、代码生成预览或 workflow 管理职责。

### 2. 目标用户

- 已有账号的终端用户。
- 需要在移动端查看个人信息、消息和少量待办信息的用户。
- 不面向后台管理员首轮复杂配置场景。

## 首轮范围

### In Scope

1. 应用壳层
   - `uniapp` 应用入口
   - 最小页面路由结构
   - 登录态恢复与登出
2. 认证能力
   - 登录
   - refresh 恢复会话
   - `me` 拉取当前身份
3. 基础页面
   - 登录页
   - 首页或个人中心页
   - 通知列表页
   - 通知详情或已读动作页
4. 最小状态能力
   - 全局 loading / empty / error 三态
   - 基础鉴权失效处理
   - 最小设备端缓存策略
5. 终端范围
   - 首轮只要求 `H5` 调试可跑通
   - 首轮只要求一种移动端真机链路可验证，优先 `Android`

### Out of Scope

1. 不做企业后台壳层
   - 不做菜单树、工作区、tabs、多模块后台路由
2. 不做复杂平台能力
   - 不做 generator、SQL preview、workflow 设计或管理页
3. 不做全端同时铺开
   - 不承诺微信小程序、支付宝小程序、鸿蒙等多端同时可用
4. 不做共享抽象前置
   - 不先抽 `Vue + uniapp` 共用 UI、共用状态层、共用 adapter
5. 不做高风险原生能力
   - 不先接推送、地图、支付、即时通讯、原生桥接

## 首轮页面建议

### P1. 登录页

- 用户名/密码登录
- 如服务端租户策略要求，可预留 `tenantCode` 输入位
- 登录成功后进入首页

### P2. 首页 / 个人中心

- 展示当前用户身份摘要
- 展示登录态是否有效
- 提供跳转通知页与退出登录入口

### P3. 通知列表页

- 拉取当前用户通知列表
- 支持已读/未读状态展示
- 支持进入详情或执行最小“标记已读”

### P4. 通知详情页

- 展示基础通知内容
- 进入详情时补做已读或提供显式已读动作

说明：

首轮选择通知而不是系统模块工作区，是因为通知天然符合 C 端“查看与处理个人消息”的使用语义，也能直接复用当前已存在的服务端 contract。

## 阶段拆分

### Phase 0: 范围冻结

- 固定首轮只做认证 + 个人中心 + 通知闭环
- 固定首轮终端只要求 `H5 + Android`
- 固定不进入后台、工作区、代码生成、workflow 管理

### Phase 1: 启动前门槛

启动实现前必须确认：

1. `uniapp` 的 app owner 是否落在 `apps/example-uniapp`
2. 是否需要独立 `packages/frontend-uniapp`
3. 首轮验证命令与真机验收路径是否明确
4. 认证 cookie / token 在 `uniapp` 环境下的承接方式是否明确

### Phase 2: 运行时最小骨架

- 搭建应用入口
- 接入最小 API 调用封装
- 接入登录态恢复
- 完成页面路由与基础错误态

### Phase 3: 首轮业务闭环

- 跑通登录
- 跑通个人中心
- 跑通通知列表、详情与已读
- 补最小手工回归脚本

### Phase 4: 第二轮候选扩展

只有在首轮稳定后才考虑：

- 文件上传/下载
- 个人资料编辑
- 单一 C 端业务表单
- 第二个业务模块

## 复用地图

### 可以直接复用

- `apps/server` 的 auth / notification / file 等 API contract
- `packages/schema` 中稳定的 DTO / schema
- `packages/ui-core` 中不携带框架语义的协议

### 只可参考，不可直接复用

- `apps/example-vue` 的页面组织方式
- `packages/frontend-vue` 的登录态处理思路

### 明确不复用

- `packages/ui-enterprise-vue` 的组件实现
- `apps/example-vue` 的后台工作区壳层

后续只有同时满足以下条件，才允许从文档进入实现：

1. `Vue` 主线不再处于高频结构波动期
2. `uniapp` owner 和目录边界已写清
3. 已明确至少一条可执行验证链路
4. 没有要求同步启动多个终端平台

## 风险

### 架构风险

- 若把 `uniapp` 首轮范围定成后台工作区，会与当前 `Vue` 主线直接重叠。
- 若把多端兼容作为首轮目标，会把业务验证问题变成端适配问题。

### 范围膨胀风险

- 很容易从“消息查看”膨胀成“完整移动工作台”。
- 很容易因为追求复用而提前抽 shared owner。

## 回滚路径

- 当前阶段只涉及文档，回滚只需回收本文件与路线图引用。
- 若后续决定不做 `uniapp`，不影响现有 `Vue` 主线和服务端 contract。

## 结论

`uniapp` 的合理首轮范围应是：

- 第二优先级
- 移动端用户自助
- 认证 + 个人中心 + 通知闭环
- `H5 + Android` 最小验证
- 不进入后台工作区、不进入多端同时铺开、不提前抽跨前端 shared
