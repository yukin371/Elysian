# 2026-05-21 GVA / RuoYi / elysia-admin 对标与 UX First 自我优化计划

> 状态：计划态
> 目标：把“对标开发”从口号收敛成可执行判断规则，既补齐竞品已有能力，也明确 Elysian 必须做出的选择理由。

## 1. 背景

当前仓库已经具备：

- 参考发行版的后台基础骨架
- 标准系统模块与多租户、workflow、generator 的最小闭环
- CI、gate、report、handoff、rehearsal 等治理链路

当前真正需要理清的，不再是“要不要对标”，而是：

1. `GVA / RuoYi / elysia-admin` 分别用来回答什么问题
2. 哪些能力属于必须补齐的下限，哪些能力属于应该做得更好的体验，哪些能力属于 Elysian 必须形成的选择理由
3. 后续功能强化应怎样保持 `UX first`，避免再次退回“先堆功能，再想用户为什么要用”

## 2. 边界

本计划只定义：

- 对标对象的使用方式
- `Parity / Better / Unique` 三层能力分类
- 后续功能强化的优先顺序
- `UX first` 的执行规则与验收入口

本计划不直接声明以下能力已完成：

- 新的后台模块
- 新的 generator 能力
- 新的部署平台
- 新的 AI 自动化基础设施

实现仍需回到各自 canonical owner：

- `apps/example-vue`：后台工作区装配与真实交互
- `apps/server`：HTTP、auth、权限、运行态接口
- `packages/persistence`：持久化真源
- `packages/generator`：schema -> preview/report/apply artifact

## 3. 对标对象分工

### 3.1 `RuoYi`

作用：

- 作为“基础后台功能下限”参考
- 用于判断系统管理、权限、日志、会话、导入导出、后台工作区是否达到企业后台的常用水位

不用于：

- 定义 Elysian 的技术结构
- 驱动 Elysian 引入 Spring 体系式的基础设施

### 3.2 `GVA`（gin-vue-admin）

作用：

- 作为“后台 starter 交付效率”参考
- 用于判断一个新模块从后台能力、菜单、权限、代码生成到接线效率是否足够顺手

不用于：

- 要求 Elysian 复刻 Go 技术栈
- 把 generator 直接扩写成“一键生成即正式接线完成”

### 3.3 `elysia-admin`

作用：

- 作为“同生态直接比较对象”参考
- 用于判断 Elysian 在 `Elysia + Bun` 语境下是否真的提供更强的后台能力、模块装配效率与工程治理能力

不用于：

- 逼迫仓库为了同生态比较而牺牲既有 owner 边界
- 把同生态已有功能机械搬运为当前主线事实

### 3.4 `v0 / Bolt / Lovable / Replit Agent / Firebase Studio`

作用：

- 作为“首屏体验、上手速度、非工程用户预期”参考
- 用于提醒 Elysian 不能只强调工程治理，还必须降低首次使用与持续交付摩擦

不用于：

- 让 AI 绕过 schema / generator 直接改核心工程
- 把黑盒平台体验误写成 repo-native 工程能力

## 4. 选择理由：用户为什么选 Elysian

用户不会因为“我们也有后台模块”就选择 Elysian。

Elysian 必须同时满足三层价值：

### 4.1 `Parity`：别人有的，我们必须有

这是入场券，不是卖点。

包括：

- 登录、权限、菜单、会话治理
- 用户、角色、菜单、部门、岗位、字典、配置
- 文件、通知、操作日志、租户
- 后台壳层、工作区切换、导出与基础验证

### 4.2 `Better`：别人也有的，我们要做得更顺

这是试用后的真实体感。

优先体现在：

- 模块可找、操作路径短、错误反馈可理解
- generator 的预览、确认、apply 与证据展示更清楚
- 启动、验证、回放、排查路径更短
- 文档、脚本、handoff 与 gate 更易执行

### 4.3 `Unique`：别人通常没有或没做扎实的，我们必须做成选择理由

这是 Elysian 不能丢的护城河。

当前最值得坚持的方向：

- `AI -> Schema -> Generator -> Handoff` 的结构化交付链路
- 生成结果进入 repo，可审查、可确认、可回滚、可人工接管
- `report / gate / evidence / rehearsal` 组成的治理能力
- 在多租户、workflow、发布演练等高风险区域保持边界与验证意识

一句话收口：

`Elysian 不是只做一个能跑的后台，而是做一个可持续交付企业模块的后台生产线。`

## 5. 对标与自我优化总表

| 层级 | 目标问题 | 主要参考 | 当前判断 | 下一步动作 |
| --- | --- | --- | --- | --- |
| `Parity` | 常见后台能力齐不齐 | `RuoYi` + `GVA` + `elysia-admin` | 仓库已具备大部分后端与工作区骨架，但部分工作区仍需继续收口为“真实后台” | 继续按已有模块清单补齐页面闭环、权限失败态、导出与日常动作一致性 |
| `Better` | 同类能力是否更顺手 | `GVA` + AI builder 体验 | generator 与治理链路已有优势，但首屏认知成本、结果判断与首次上手仍需继续压缩 | 保持 `demohub -> 正式页` 的 UX 验证顺序，持续清理研发术语、重复摘要与多余说明 |
| `Unique` | 用户为什么不直接选别的后台 | Elysian 自身主张 | 结构化 schema/generator/handoff 与治理证据是当前最强差异化方向 | 强化“从需求到可审查产物”的主线体验，不把优势淹没在纯功能对齐里 |

## 5.1 当前纠偏：不要重复盘点“已经有的基础功能”

现阶段对标开发最容易犯的错误，是继续围绕“用户、角色、菜单、文件、日志这些基础项是否存在”重复做功能盘点。

这类盘点在当前仓库里已经不再是主问题，因为：

- 大部分基础模块后端已经闭环
- 多数关键工作区已经有对应前端入口
- 多租户、会话治理、workflow、generator 与发布演练等高价值能力也已具备最小形态

因此，后续对标不再优先回答：

- `有没有这个模块`
- `有没有这个接口`
- `有没有这个按钮`

而优先回答：

- `默认用户路径是否足够短`
- `新模块落地路径是否足够顺`
- `结果是否足够可判断、可审查、可接手`
- `同生态或同类后台里，用户为什么更愿意继续选 Elysian`

一句话纠偏：

`后续对标要从“功能是否存在”切到“现有能力是否已经形成更强的默认路径和交付体验”。`

## 5.2 优秀对标设计码

这里的“设计码”不指 UI 外观，而指值得借鉴的默认交互路径、模块落地路径和用户判断路径。

| 对标对象 | 已验证值得参考的设计码 | 不该机械照搬的部分 | Elysian 应落的点 |
| --- | --- | --- | --- |
| `RuoYi` | 系统管理模块齐全、菜单/权限/日志/导出路径明确，用户进入后台后很快知道去哪里做什么 | Spring 体系结构、历史包袱和平台实现细节 | 后台主工作区的首屏任务感、模块可找性、导出与高频动作的一致性 |
| `GVA` | 代码生成与后台模块接线强调“做一个模块要快”，菜单、权限、列表、表单、接口组织更偏交付效率 | Go 技术栈与其内部实现方式 | `AI -> Schema -> Generator -> Handoff` 这条链必须比“手工接线后台”更短、更清楚 |
| `elysia-admin` | 同生态里更强调模块自动注册、Elysia/Bun/Drizzle 语境下的快速启动与开发者友好度 | 为了追求自动化而牺牲 review、handoff、owner 边界 | 要证明 Elysian 不只是“也能用 Elysia 做后台”，而是能把生成、治理和人工接手做得更完整 |
| AI builder | 首屏进入任务快、第一次产出快、非工程用户不需要先学习系统结构 | 黑盒平台体验、弱 repo-native 资产控制 | 新用户从“想试试”到“真的跑起来”的路径必须继续缩短 |

## 5.3 下一轮对标的真正缺口

结合当前仓库现状，下一轮不再把以下内容当作主要缺口：

- `user / role / menu / department / file / notification / log` 是否存在
- 基础登录、RBAC、session、tenant 能力是否存在
- 是否已经有 generator、preview、apply 与 CI gate

下一轮真正值得继续追的缺口应改为：

1. `工作区默认路径`
   - 用户进入模块后，首屏是否先服务任务
   - 查询、列表、详情、编辑、导出、危险动作是否形成一致节奏

2. `新模块落地效率`
   - 起稿方式是否容易理解
   - 结果是否能直接判断影响面
   - handoff 后人工接线是否足够清楚

3. `可判断性`
   - 冲突、拒绝、漂移、权限不足、空状态、失败恢复是否都告诉用户下一步

4. `同生态选择理由`
   - 与 `elysia-admin` 相比，Elysian 是否能明显展示更强的 repo-native 资产、治理证据与人工接手能力

## 6. 功能强化优先顺序

后续功能强化默认按以下顺序执行。

### 6.1 第一顺序：先补 `Parity`

优先收口：

- `apps/example-vue` 已有系统工作区的真实后台可用性
- 查询、列表、详情、创建、编辑、删除、状态动作、导出的一致性
- 登录、权限、会话、日志、文件、通知、租户等高频后台体验

判断规则：

- 如果竞品已有且用户高频感知，这类能力先补齐
- 若已有后端契约，只先补前端工作区闭环
- 若契约不足，再按 canonical owner 补最小接口或持久化能力

### 6.2 第二顺序：再打磨 `Better`

优先打磨：

- 首次进入后台后的找模块路径
- 标准 CRUD 的首屏任务感
- generator preview 的起稿、结果判断、审查、apply 路径
- 错误、漂移、拒绝、无权限、空状态的下一步建议

判断规则：

- 一个功能如果用户已经能完成，但过程仍重、绕、难判断，就进入这一层
- 这一层优先删噪音、缩路径、补证据，而不是先加新模块

### 6.3 第三顺序：最后放大 `Unique`

优先方向：

- `AI -> Schema -> Generator -> Handoff` 的结构化体验继续收口
- review / confirm / apply / evidence 的用户可读性持续增强
- 生成后人工接线完成证据、回放能力与审计链继续清晰化

判断规则：

- 新能力必须增强 Elysian 的选择理由，不能只是“多一个按钮”
- 任何差异化能力都不得破坏 `schema / generator / persistence / runtime` 的 owner 边界

## 7. UX First 执行规则

后续所有“对标开发”默认都要先过这组问题。

### 7.1 功能判断顺序

1. 这个能力属于 `Parity`、`Better` 还是 `Unique`
2. 它主要对标谁：`RuoYi`、`GVA`、`elysia-admin`，还是 AI builder 体验
3. 它能否让用户更快完成当前任务，而不是只多一个功能点
4. 它是否已经有明确 owner、验证方式和失败路径

### 7.2 页面判断顺序

1. 用户来到这个页面的第一步是什么
2. 首屏是否先展示这一步，而不是先解释系统
3. 当前页面是否仍在使用研发态术语、重复摘要或无关说明
4. 当前错误是否告诉用户下一步该改什么
5. 当前结果是否足以支持“继续 / 通过 / 拒绝 / 应用”决策

### 7.3 文档判断顺序

1. 文档是否明确区分“已实现事实”和“下一步计划”
2. 文档是否说明该能力为什么属于 `Parity / Better / Unique`
3. 文档是否说明验证方式、风险与 non-goals

## 8. 近期工作包

### WP-1 对标矩阵归一化

目标：

- 形成统一的对标判断表，不再把 `RuoYi / GVA / elysia-admin / AI builder` 混写

交付：

- 当前计划文档
- 模块级对标矩阵
- `roadmap` 支持性轨道引用
- UX BDD 规格中的价值判断与参考对象同步

对应文档：

- [2026-05-21-module-level-benchmark-matrix.md](./2026-05-21-module-level-benchmark-matrix.md)

### WP-2 参考发行版 `Parity` 收口

目标：

- 让 `apps/example-vue` 更像完整、真实、可操作的后台

优先范围：

- `users / roles / menus / departments / posts / dictionaries / settings / operation-logs / notifications / files / tenants`

验收重点：

- 模块可找
- 主任务直接可见
- 空态、失败态、权限态可理解
- 导出与高频动作的作用范围清楚

### WP-3 Generator 的 `Better + Unique` 收口

目标：

- 让 generator 不只“能生成”，而是“让用户敢判断、敢接手、敢继续”

优先范围：

- 起稿入口
- 最近结果 / 生成结果的主次关系
- diff / SQL proposal / blocker / recovery / apply evidence 的可读性
- 拒绝、漂移、重试与人工接管路径

### WP-4 首次上手与交付体验

目标：

- 降低新用户从“想试试”到“真的跑起来”的摩擦

优先范围：

- 最短启动路径文档
- 验证路径文档
- 环境说明
- 常见失败与恢复路径

## 9. 非目标

当前不把以下方向写成这一轮对标开发的完成定义：

- 复刻任一竞品的技术栈
- 新增第二套 starter 或第二套 shared owner
- 把 generator 扩成自动正式 migration / 自动正式模块接线
- 把导入平台、低代码设计器、SQL 工作台、多通道通知中心一并拉进当前主线
- 为了“看起来更全”而牺牲边界、验证和可维护性

## 10. Definition of Ready

任何后续“对标开发”任务在进入实现前，都至少要写清：

```text
能力层级：Parity / Better / Unique
主要对标对象：
目标用户：
目标模块：
现有 owner：
当前缺口：
最小改动：
验证方式：
非目标：
```

## 11. Definition of Done

完成任一对标强化任务后，都至少输出：

```text
已完成改动：
验证结果：
未验证区域：
同步文档：
残留风险：
```

并额外检查：

1. 是否补齐了用户高频感知的能力下限
2. 是否缩短了用户完成任务的路径
3. 是否强化了 Elysian 的选择理由，而不是只多了一个对标功能点
4. 是否仍然守住 canonical owner 与验证边界
