# Phase 2 Auth Foundation Plan

更新时间：`2026-04-21`

> 说明：本文件是短期实施计划，只描述本轮建议拆分与任务顺序。实际推进状态统一以 [../roadmap.md](../roadmap.md) 为准。

## 目标

把 `Elysian` 从“只有业务垂直切片”推进到“具备统一身份、权限、菜单和最小审计能力”的平台底座状态。

## 范围

本计划对应主计划中的：

- `P2A` 认证协议定稿
- `P2B` 身份与 RBAC 模型
- `P2C` 登录与权限接入
- `P2D` 审计基线

## 本轮不做

- OAuth / SSO / LDAP
- 复杂组织级数据权限
- 多端统一登录态同步
- 审计日志高级检索与导出

## 工作包拆分

### WP-2A-1 认证 ADR

- 输出：一份 ADR，固定 JWT、refresh token、失效策略、前端持久化方式
- 关注点：
  - access token 生命周期
  - refresh token 是否持久化
  - logout 是前端删除还是服务端失效化
  - 401 / 403 / token expired 的错误语义
- 验收：后端接口与前端存储策略不再反复摇摆

### WP-2B-1 RBAC 数据模型

- 输出：`user / role / permission / menu` 表结构草案
- 参考文档：[2026-04-21-phase-2-rbac-data-model.md](./2026-04-21-phase-2-rbac-data-model.md)
- 关注点：
  - 权限点命名规范
  - role 与 permission 的关系模型
  - menu 与 permission、route 的映射方式
  - seed 数据是否覆盖 admin / operator 两类角色
- 验收：能支持最小登录态和动态菜单返回

### WP-2B-2 最小服务端骨架

- 输出：`auth / user / role` 模块骨架和迁移草案
- 关注点：
  - 中间件挂载点
  - user context 注入方式
  - 权限守卫是函数式还是装饰器式
- 验收：server 侧能表达“已登录”和“有权限”两个基本概念

### WP-2C-1 登录链路

- 输出：`login / refresh / logout / me` 最小接口
- 关注点：
  - 密码校验与加密
  - token 签发与刷新
  - 未认证请求返回 401
- 验收：真实账号可登录、刷新、登出

### WP-2C-2 前端权限接入

- 输出：登录页、路由守卫、动态菜单、权限组合式函数
- 关注点：
  - 未登录跳转
  - 启动时恢复登录态
  - 动态菜单与本地路由注册的边界
- 验收：不同角色看到不同菜单和页面

### WP-2D-1 审计基线

- 输出：登录、刷新、登出、权限拒绝审计记录；后续权限变更模块接入同一审计表
- 关注点：
  - 记录字段是否覆盖 actor / action / target / result
  - 是否需要 request id 和 ip
  - 在尚未落用户 / 角色管理模块前，不把“权限变更”写成已实现事实
- 验收：关键权限动作不再是黑盒

## 建议顺序

1. 先写 `WP-2A-1`，不先定协议就不要写登录代码。
2. 再做 `WP-2B-1` 和 `WP-2B-2`，先把模型和中间件接入点定下来。
3. 之后做 `WP-2C-1` 和 `WP-2C-2`，打通登录和前端接入。
4. 最后补 `WP-2D-1`，把关键行为收口到审计体系。

## 完成标准

- 有一份已确认的认证 ADR
- 有可运行的 user / role / permission / menu 数据模型
- 登录后可以拿到 token 并访问受保护接口
- 未认证请求返回 401，无权限请求返回 403
- 前端能根据菜单和权限控制页面展示
- 关键操作可产生审计记录

## 依赖

- 依赖 Phase 1 已完成的 server / persistence / frontend 基线
- 依赖后续 ADR 固定密码加密与 token 签名方案

## 风险

- 如果先写实现再定认证 ADR，后续回改成本会很高
- 如果权限点命名过于随意，Phase 4 的 generator 将很难自动生成
- 如果菜单和路由模型绑定过死，后续 React 适配层会被拖累
