# 2026-05-21 模块级对标矩阵（路径 / 交付 / 体验）

> 状态：计划态
> 用途：把“对标开发”从功能名对照推进到模块级默认路径、交付效率与结果判断矩阵。

## 1. 使用方式

本矩阵不再回答“这个模块有没有”。

本矩阵只回答：

1. 某条默认路径是否已经足够像成熟后台 / starter
2. 当前缺口是在能力本身、路径设计、交付效率，还是结果可判断性
3. 这条缺口属于 `Parity`、`Better` 还是 `Unique`
4. 后续应该由哪个 owner 继续推进，以及怎么验证

## 2. 优先级说明

- `P0`：已经有能力，但默认路径仍会直接影响“像不像完整后台”或“敢不敢继续用”
- `P1`：已有最小闭环，但还没形成更强的交付体验或判断体验
- `P2`：可作为后续增强，但当前不应抢主线

## 4. 当前实施顺序

### 第一步：先做 `P0`

1. 后台壳层进入路径
2. 模块可找性与导航路径
3. 标准 CRUD 列表路径
4. 导出路径
5. Generator 起稿路径
6. Generator 结果判断路径
7. Generator apply / handoff 路径
8. 新模块交付效率

### 第二步：再做 `P1`

1. 操作日志定位路径
2. 通知工作区路径
3. 文件管理路径
4. 在线会话治理路径
5. 同生态选择理由
6. 首次上手路径

### 第三步：保留 `P2`

- 当前矩阵还没有单列 `P2` 项；后续若继续扩展导入、对象存储治理、复杂风控、多环境 apply，再另行补入，不提前抢当前主线

## 5. 使用要求

后续要从本矩阵落任务时，默认动作不是“新建更多能力项”，而是：

1. 先选一条 `P0 / P1` 路径
2. 明确该路径属于 `Parity / Better / Unique`
3. 只补最小缺口
4. 明确验证方式
5. 完成后回填当前矩阵、`roadmap` 或相关 BDD / 计划文档

## 6. 一句话结论

`Elysian 现在真正要追的，不是再证明基础功能存在，而是把已有能力收口成更短的默认路径、更强的结果判断和更明确的选择理由。`
