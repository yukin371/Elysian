# DEVELOPMENT PRINCIPLES

更新时间：`2026-04-27`

本文件定义仓库在当前阶段的开发与审查原则，用于约束 AI 和人工协作时的默认决策顺序。

## 核心结论

`Clean Code` 和 `SOLID` 可以使用，但不能高于仓库的边界治理目标。

本仓库当前更高优先级的原则是：

1. 先边界，后实现
2. 先验证，后扩展
3. 先复用，后新增
4. 先契约，后技巧

若局部“代码更优雅”和整体“owner 更稳定”冲突，优先选择后者。

## 优先级顺序

### 1. Boundary First

先确认模块边界、canonical owner、依赖方向、验证方式，再开始实现。

适用要求：

- 不为了复用而跨 owner 抽 shared helper
- 不为了“结构更漂亮”而把运行时责任拆散到多个层
- 不为了“统一风格”改写当前阶段尚未稳定的边界

### 2. KISS

优先最直接、最少概念、最少抽象的实现。

适用要求：

- 能用现有模块完成，就不要新增包装层
- 能用直接数据流表达，就不要先引入复杂模式
- 能用小范围重复保持清晰，就不要为避免重复而过早抽象

### 3. YAGNI

当前阶段未被验证链路证明必要的能力，不提前实现。

适用要求：

- 不预留未进入主线的重型基础设施
- 不为未来可能存在的扩展点提前建立接口层级
- 不把当前最小闭环扩写成通用平台能力

### 4. High Cohesion, Low Coupling

让模块内部职责集中，对外依赖最少且明确。

适用要求：

- 让能力停留在自己的 canonical owner 内
- 避免无边界的 `utils`、`helpers`、`shared` 桶文件
- 避免调用方向反转到违反 `ARCHITECTURE_GUARDRAILS` 的程度

### 5. Explicit Contracts

输入、输出、错误、权限、状态、持久化语义应尽量显式。

适用要求：

- 优先稳定 schema / DTO / response envelope / permission code
- 优先写清验证命令与测试入口
- 优先补“对外行为”测试，而非隐藏实现细节

### 5.5. Design By Document

前端页面的视觉与文案规则必须显式落在 `DESIGN.md`，而不是停留在个人偏好或口头约定。

适用要求：

- 新增或修改前端页面前，先确认该页面受哪个 `DESIGN.md` 约束
- 若目标模块已有局部 `DESIGN.md`，必须在共享 `DESIGN.md` 之上继续遵守局部规则
- 一般情况不要求每个前端模块单独补 `DESIGN.md`；若上层设计文档已足够，默认直接继承
- 若页面已经形成稳定且独立于上层规则的风格约束，再补局部 `DESIGN.md`，不要把它混写进 `MODULE.md`
- 不允许把研发态措辞、颜色体系、圆角体系和容器层级长期留在“实现里自己看”

### 6. Composition Over Inheritance

优先组合现有能力，不引入深层继承或过强耦合的抽象树。

### 7. DRY After Proof

只有在重复模式已经稳定、且不会制造错误 owner 时，才做提炼。

适用要求：

- 两三处重复并不自动等于必须抽象
- 若抽象会模糊模块边界，宁可保留局部重复
- 若抽象会降低可读性、验证性或回滚性，不应提取

### 8. Test The Boundary, Not The Trick

优先测试模块边界、权限边界、持久化边界、阶段门禁，不优先测试纯实现技巧。

### 9. Document Decisions, Not Trivia

记录长期有效的边界、owner、约束、验证入口和阶段结论；不要把短期实现噪音写成长期规则。

### 10. Keep Files Reviewable

默认保持文件体量可审查、可拆分、可回滚，避免单文件长期超过 `1000` 行。

适用要求：

- 不把多个工作区、多个页面分支、多个纯逻辑集合长期堆在同一文件
- 文件接近 `1000` 行时，优先审查是否可以在当前 owner 内做受控拆分
- 拆分优先落在原 owner 内部，不借机抽新的 shared `utils`、`helpers`、`base` 文件
- 若暂时必须超过 `1000` 行，应补充原因和后续拆分计划，而不是把超限视为常态

## Clean Code 和 SOLID 的使用方式

### 可以用的部分

- 用 `SRP` 检查职责是否过宽
- 用 `ISP` 检查接口是否强迫不相关依赖
- 用 `DIP` 检查依赖方向是否稳定
- 用 `Clean Code` 检查命名、重复、可读性、函数长度和错误语义

### 不应机械套用的部分

- 不为了追求 `OCP` 而提前抽插件点、策略层、抽象基类
- 不为了追求“零重复”而制造共享桶文件
- 不为了“函数短小”把连续业务语义切碎
- 不为了“解耦”把简单链路拆成难追踪的间接层

## 对 AI 的明确约束

AI 在实现和审查时，默认按以下顺序决策：

1. 这次改动是否符合当前阶段边界
2. 这次改动是否落在正确 owner
3. 是否已有现有实现可复用
4. 是否有更简单、更少抽象的做法
5. 是否有明确验证方式
6. 若改动涉及前端页面，是否已经对照对应 `DESIGN.md`
7. 最后才讨论代码是否足够“优雅”

AI 不应做的事：

- 为了“clean”主动扩大重构范围
- 为了“SOLID”新增当前阶段不需要的接口、基类、适配器
- 在缺少验证链路时大规模整理代码结构
- 用风格统一名义跨模块搬运职责
- 把规划态能力写成已实现事实
- 在文件已经明显超限时继续无计划堆叠实现，而不先评估受控拆分路径

## 审查发生时机

这套原则不应只在代码写完后才使用，默认应在以下时点发生审查：

### 1. 开发前自审

开始改动前，先审：

- 这次改动是否符合当前阶段边界
- 这次改动是否落在正确 owner
- 是否已有现有实现可复用
- 验证方式是否明确

### 2. 新增抽象前复审

当准备新增 helper、adapter、service、shared util、接口层或重构公共抽象时，再审一次：

- 这次抽象是否真的必要
- 是否会形成重复 owner
- 是否只是为了追求 `Clean Code` / `SOLID`
- 是否会扩大当前任务范围

### 3. 提交前自审

提交前至少再审一次：

- 验证是否完成
- 文档是否需要同步
- 风险与回滚路径是否明确
- 是否存在未验证区域

### 4. PR 正式审查

进入 Pull Request 后，由 reviewer 按边界、验证、风险和文档同步进行正式审查。

### 5. 合并前门禁审查

CI、分支保护、必填模板和验证命令共同构成最后一道审查。

## 审查清单

在提交实现或 review 时，至少检查：

1. 是否守住当前主线边界，而不是顺手扩大范围
2. 是否复用现有 owner，而不是新增重复实现
3. 是否存在过早抽象、过早通用化、过早基础设施化
4. 契约、权限、错误和验证方式是否足够显式
5. 测试是否覆盖边界行为，而不是只覆盖内部实现
6. 文档是否只同步真正发生变化的长期事实
7. 若涉及前端页面，是否已对照并同步对应 `DESIGN.md`

## 一句话规则

`先边界，后抽象；先验证，后扩展；先复用，后新增；先契约，后技巧。`
